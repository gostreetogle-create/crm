# Чеклист: полное подключение массового импорта JSON (новая сущность)

Цель: **досконально** провести одну таблицу/справочник от спецификации до рабочего UI в «Админ-настройках», согласованно с **доками**, **HTTP API**, **матрицей прав** и **фронтом**.

**Связанные документы**

| Документ | Роль |
|----------|------|
| [`dev-bulk-json-migration-checklist.md`](./dev-bulk-json-migration-checklist.md) | Историческая миграция Excel → JSON, аудит `xlsx`, права `excel.*`. |
| [`frontend/rbac-and-admin-settings.md`](./frontend/rbac-and-admin-settings.md) | Как добавлять `PermissionKey`, каталог, матрицу. |
| [`frontend/dictionary-field-behavior-guide.md`](./frontend/dictionary-field-behavior-guide.md) | Канон полей справочников и bulk. |
| [`frontend/new-dictionary-checklist.md`](./frontend/new-dictionary-checklist.md) | Новая плитка на хабе (если сущность новая). |
| `backend/prisma/schema.prisma` | Источник правды по полям БД. |

**Текущее состояние репозитория (ориентир):**

- Рабочий bulk только у **единиц**: `POST /api/bulk/units` (роутер под цепочкой `requireAuth` + `requireAdmin` в `backend/src/app.ts`, префикс **`/api`**, не `/api/admin`).
- UI: `crm-web/libs/settings-feature/.../bulk-json-import-card/` — реестр `bulk-json-import-card.targets.ts` (много строк «планируется»).
- Сервис: `BulkJsonAdminService.importBulk(segment, items)` → `POST .../api/bulk/:segment`.

Использование: выполняйте **по фазам**; для каждой новой сущности — **полный проход** или явно пометьте отложенные пункты.

## Фаза 0. Спецификация и контракт (до кода)

- [ ] **0.1** Зафиксировать **имя сущности** (латиница, slug для URL: `colors`, `production-work-types` и т.д.).
- [ ] **0.2** Сверить поля с **`schema.prisma`**: какие поля обязательны, какие FK, уникальные индексы, значения по умолчанию.
- [ ] **0.3** Решить режим: **только create** vs **upsert по `id` / естественному ключу** — зафиксировать в коротком фрагменте в `docs/frontend/*-crud.md` или отдельном `docs/frontend/bulk-<entity>.md`.
- [ ] **0.4** Описать тело запроса: **`{ "items": [ ... ] }`**, лимит размера/числа элементов (как у единиц: до 5000 или свой лимит).

- [ ] **0.7** Ошибки валидации Zod → **400** с `details` (как в `bulk-units.routes.ts` через `flatten()`), не «голый» 500.

## Реестр целей во фронте

Все строки из **`BULK_JSON_TARGETS`** должны либо иметь **`hasEndpoint: true`** и рабочий **`submit()`**, либо оставаться заглушками с пометкой в **`methodLine`** / **`detailHint`**, чтобы не создавать ожидание рабочего API. При добавлении backend для очередной строки — **обязательно** пройти фазы **1–4** для этой строки.

## Контрольный список «ничего не забыть»

- [ ] Путь API в доке = фактический (`/api/bulk/...`, не устаревший `/api/admin/bulk/...`, если в коде иначе).
- [ ] Право в матрице = проверка в UI; по возможности = проверка на сервере.
- [ ] Пример JSON в UI = проходит Zod на бэкенде.
- [ ] `docs/backend-map` + скрипт генерации overview.
- [ ] Нет расхождения между `label` в выпадающем списке и реальным именем таблицы в БД — в подсказках использовать понятные названия + slug в `methodLine`.
