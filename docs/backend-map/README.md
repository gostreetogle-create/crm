# Название файла: .md — Backend map (справочники + сущности)

Каноничное хранилище **аналитической** карты данных: малые перечисления и сущности с полями. Это не замена Prisma/OpenAPI — это слой «что строим» до и параллельно коду.

---

## Точка входа для просмотра (таблицы)

**Визуальный обзор** всех справочников и сущностей в виде Markdown-таблиц: **[`OVERVIEW.generated.md`](./OVERVIEW.generated.md)**.

- Файл **не редактируется вручную** — он собирается из JSON.
- После любого изменения в `*.json` этой папки выполни из корня репозитория:

```bash
node scripts/generate-backend-map-overview.cjs
```

Любой новый `*.json` в `docs/backend-map/` **рекурсивно** (включая вложенные папки, например `material_geometry_new_model/`) подхватывается автоматически: блоки `dictionaries` и `entities` рендерятся таблицами; прочие корневые ключи (вне стандартного набора) — в разделе «Дополнительные JSON-файлы».

---

## Зачем две части

| Часть | Файл | Смысл |
|--------|------|--------|
| Кирпичики | `small_dictionaries.json` | Конечные наборы значений со **стабильным** `key` (статусы, типы движений, ключи прав). Менять `key` после релиза нельзя без миграции данных. |
| Дом | `entities_from_dictionaries.json` | Таблицы/сущности и поля. Поля, которые берут значение из перечисления, ссылаются на справочник в `description` (`ref → dictionaries.<имя>`) или по имени (`*Key`, `*Id`). |

Один «большой» смысл модели = **справочники + сущности**; физически разнесено, чтобы не смешивать enum-листы с широкими таблицами.

---

## Конвенции

### `small_dictionaries.json`

- Корень: `version`, `dictionaries`.
- `dictionaries.<имя>`: `rus`, `description`, `items[]`.
- `items[]`: обязательно `key`, `rus`; опционально `description`.
- Имя справочника — `snake_case` (например `proposal_status`).

### `entities_from_dictionaries.json`

- Корень: `version`, `entities`, `_open_questions`.
- `entities.<id>`: `name` (рус.), `description` (домен), `fields[]`.
- Поле: `key`, `rus`, `type`, `required`, `description`.
- Типы: `uuid`, `string`, `int`, `float`, `boolean`, `date`, `datetime`, `json` — логические; маппинг в SQL решается при миграции.
- `_open_questions` — вопросы к продукту/владельцу до зафиксированной схемы.

### JSON без комментариев

Вся поясняющая «самодокументация» дублируется в `_meta` внутри каждого JSON и в этом README.

## Как обновлять (поток работы)

1. Входящие (Excel, CSV, текст) кладутся в `analysis_hub/inbox/`.
2. Разбор: новые перечисления → `small_dictionaries.json`; новые таблицы/поля → `entities_from_dictionaries.json`.
3. Дубликаты: не плодить второй справочник с тем же смыслом — расширить существующий или явно описать merge здесь.
4. После **полного** разбора файл из `inbox` удаляется; незавершённый — остаётся, в отчёте помечается «не проанализирован до конца».
5. При существенном изменении схемы поднять `version` в затронутом JSON (минор: +поля/справочники).
6. Запустить `node scripts/generate-backend-map-overview.cjs` и закоммитить обновлённый `OVERVIEW.generated.md` вместе с JSON.
