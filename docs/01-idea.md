# Общая идея проекта (v1.0)

## Назначение
CRM — это единая система справочников и данных, где:
- справочники живут отдельно (как чистые источники правды);
- карточки товаров/записей хранят только ссылки на справочники (ID), а человекочитаемый текст формируется при экспорте;
- любые изменения структуры и схем делаются через миграции, чтобы не терять труд.

## Подход к созданию текущего приложения (название раздела)
Чтобы создать и развивать текущее приложение без хаоса, мы комбинируем следующие способы разработки:

### 1) Итеративная сборка от MVP
Сначала делаем минимально работоспособный “скелет” (MVP), затем по циклам улучшаем:
- пользовательские сценарии;
- качество данных (валидность, ссылки, единообразие);
- удобство экспорта/импорта.

### 2) Контрактная архитектура данных (справочники отдельно)
Разделяем ответственность:
- “Справочники” — только уникальные сущности без дублей.
- “Данные товаров/записей” — только ссылки (ID) на справочники.
Так мы уменьшаем вероятность несогласованных правок и упрощаем миграции.

### 3) Изменения через миграции (не теряем труд)
Любая правка, которая затрагивает привязанные данные или структуру, описывается как миграция:
- что меняем;
- как переносим существующие данные;
- как проверяем корректность.
Это позволяет безопасно “переделывать всё”, не теряя уже сделанную работу.

### 4) Экспорт/импорт как человекочитаемый контракт
Для обмена данными используем формат, который удобно читать человеку:
- экспорт в Excel без ID (по текстовым значениям);
- импорт обратно через корректную обратную маппинг-логику.
Экспорт/импорт рассматриваем как публичный “контракт” процесса.

## Принцип “документация как механизм безопасности”
Документация фиксирует:
- правила (что считаем истиной);
- ограничения (что нельзя);
- порядок действий (что делать при изменениях).
Это снижает риск регрессий при подключении нового разработчика или при возврате к проекту через месяцы.
