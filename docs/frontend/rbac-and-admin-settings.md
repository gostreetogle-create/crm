# Роли, права, «Настройки» и «Админ-настройки»

> **Актуально после интеграции API и вынесения кода в `libs/`:** канон RBAC, матрица, дефолты по коду роли, санитизация GET/PUT, порядок `ensureRolesLoaded` → `syncMatrixFromServerSafe` и связанные файлы — в [**`authz-roles-architecture.md`**](./authz-roles-architecture.md) и [**`authz-matrix-runbook.md`**](./authz-matrix-runbook.md). Ниже — исторический снимок (часть путей вида `crm-web/src/app/core/auth/...` устарела; реализация сейчас в `@srm/authz-core`, `@srm/authz-runtime`, `@srm/dictionaries-state`).

Документ описывал **канон** для RBAC на фронте в ранней фазе. Цель: один согласованный поток (справочник ролей → матрица → сессия), без «зашитых» ролей кроме суперадмина.

## Где единый источник правды

| Файл | Назначение |
|------|------------|
| `crm-web/src/app/core/auth/authz.types.ts` | `RoleId`, `PermissionKey`, `CrudPermissions`. Новый ключ права — **сначала** сюда (union), затем каталог и матрица. |
| `crm-web/src/app/core/auth/authz.catalog.ts` | `PERMISSION_CATALOG` (подписи RU + группа), `PERMISSION_KEYS_ORDERED`, секции UI матрицы `AUTHZ_MATRIX_UI_SECTIONS`, `permissionKeysForAuthzGroup()`. |
| `crm-web/src/app/core/auth/dict-hub-permissions.ts` | Соответствие `tileKey` плитки хаба → `dict.hub.*`. Список всех ключей плиток: `DICTIONARY_HUB_PERMISSION_KEYS`. |
| `crm-web/src/app/core/auth/authz.matrix.ts` | `DEFAULT_ROLE_PERMISSIONS_BY_CODE`: **только** код `admin` (полный набор ключей). Любой другой код без записи → пустой набор до настройки матрицы. Устаревший алиас `ROLE_PERMISSIONS` — не опираться на него в новом коде. |
| `crm-web/src/app/core/auth/permissions.service.ts` | Текущая роль сессии (`localStorage` `crm.currentRole`, значение = **id** роли из справочника), эффективные права, переопределение матрицы (`crm.authz.matrixOverride`), `can` / `hasAny`, `crud`, суперадмин вне матрицы. |
| `crm-web/src/app/features/roles/**` | Модель `RoleItem`, seed, `ROLES_REPOSITORY`, mock-репозиторий, `RolesStore`. Справочник «Роли» на `/dictionaries`. |
| `crm-web/src/app/features/users/**` | Справочник «Пользователи»: `UserItem`, `USERS_REPOSITORY`, `UsersStore`, плитка `tileKey="users"`, ключ `dict.hub.users`. Детали: [`users-dictionary.md`](./users-dictionary.md). |
| `crm-web/src/app/app.config.ts` | Провайдеры `ROLES_REPOSITORY` и `USERS_REPOSITORY` → mock-репозитории (до API). |
| `crm-web/src/app/core/auth/permission.guard.ts` | Защита маршрутов по `route.data.permission`. |
| `crm-web/src/app/app.routes.ts` | `data: { permission: 'page....' }` + `permissionGuard`. |

Кнопки и действия в карточках справочников завязаны на `crud.*` и `excel.*` через `permissions.crud()` / `permissions.can(...)` и при необходимости `*appHasPermission`.

## Модель данных (важно не путать)

| Понятие | Где живёт | Комментарий |
|---------|-----------|-------------|
| **Роль** | Справочник «Роли» (`RoleItem`: `id`, `code`, `name`, `sortOrder`, …) | `sortOrder` — целые **1, 2, 3…**: порядок колонок в матрице и списков (**меньше** = левее / выше). При создании и дублировании в UI подставляется **следующий** номер (max + 1). Поле **`code`** в карточке роли **не вводится**: задаётся при первом сохранении из названия (латинский slug, уникальный), дальше **не меняется** при переименовании — нужно для экспорта ролей и колонки «Код роли» в импорте пользователей. Матрица и сессия завязаны на **`id`**. На плитке хаба без колонки «Пор.» — только название. |
| **Суперадмин** | Ровно одна роль с `isSystem: true` (seed: «Администратор») | Не удаляется; **всегда** все `PermissionKey`; колонки в матрице **нет**; запись матрицы по её `id` в `localStorage` игнорируется. |
| **Текущая роль сессии** | `crm.currentRole` | Хранится **id** роли. Легаси-значения `admin` / `editor` / `viewer` при чтении мапятся на стабильные id из seed. |
| **Матрица прав** | `crm.authz.matrixOverride` | Объект: ключ = **id** роли, значение = массив `PermissionKey`. Перекрывает пустое умолчание для этой роли. Суперадмин в объект не пишем. |

Код `admin` **зарезервирован** только для суперадмина. В Excel импорта ролей колонку «Код» можно **оставить пустой** — код сгенерируется из названия; если указать вручную — те же правила, что раньше (латиница, уникальность, не `admin`).

## Группы ключей `PermissionKey`

1. **`page.*`** — доступ к маршрутам / разделам (в т.ч. `page.admin.settings`, `page.preferences`).
2. **`dict.hub.*`** — видимость **плитки** на `/dictionaries` (ключ на плитку). Новая плитка: добавить ключ в `authz.types.ts`, `PERMISSION_CATALOG`, `dict-hub-permissions.ts` (`TILE_TO_PERMISSION` + `DICTIONARY_HUB_PERMISSION_KEYS`), обернуть плитку в `@if (permissions.can(hubTilePerm('tileKey')))`.
3. **`crud.*`** — создание / правка / удаление / дублирование строк в карточках.
4. **`excel.*`** — шаблон, импорт, экспорт в панели плитки.

Секции матрицы в UI задаются `AUTHZ_MATRIX_UI_SECTIONS` в `authz.catalog.ts` (четыре блока: страницы, справочники на хабе, кнопки в карточках, Excel).

## Поведение до API

1. **Список ролей** — из `RolesStore` (источник — `ROLES_REPOSITORY`). Новая роль появляется в **Админ-настройках** автоматически (колонка матрицы), если роль **активна** и **не** `isSystem`.
2. **Суперадмин** — не настраивается галочками: `effectiveKeysForRole` всегда возвращает полный `PERMISSION_KEYS_ORDERED`.
3. **Остальные роли** — без строки в `crm.authz.matrixOverride` для этого `id` действует **пустой** набор прав (ничего не отмечено в каноне кроме исторического смысла кода: в `DEFAULT_ROLE_PERMISSIONS_BY_CODE` больше нет пресетов для `editor` / `viewer` и т.д.). Права задаются только матрицей в браузере (до API).
4. **«Сбросить галочки» по колонке** — явный пустой набор прав для роли (все галочки сняты), не возврат к дефолту по коду. **«Сбросить все переопределения матрицы»** — очищает весь объект; кнопка по колонке **неактивна**, если галочек уже нет.
5. После бэкенда: заменить источник ролей и матрицы на API; **ключи `PermissionKey` не переименовывать** без миграции — меняется только источник данных.

## Чеклист: новое **право** (новый `PermissionKey`)

1. Добавить литерал в `PermissionKey` в `authz.types.ts`.
2. Добавить строку в `PERMISSION_CATALOG` (подпись RU + `group`: `page` | `dict` | `crud` | `excel`).
3. Добавить ключ в `PERMISSION_KEYS_ORDERED` в `authz.catalog.ts` (логичное место в списке).
4. Если это плитка хаба — обновить `dict-hub-permissions.ts` и разметку `dictionaries-page` (образец: плитки «Роли», «Пользователи»).
5. Повесить проверку в UI (guard маршрута, `permissions.can`, входы `CrudLayout`, и т.д.).
6. Обновить этот документ, если меняется смысл канона.

## Чеклист: новая **роль** (бизнес-роль пользователя)

Роль в коде **не** добавляется как enum. Достаточно:

1. Создать запись в справочнике «Роли» на `/dictionaries` (или seed при первом деплое).
2. В **Админ-настройках** выставить галочки для новой колонки.

Не дублировать список ролей в `authz.catalog.ts` — он был удалён намеренно.

## Маршруты и UI

| Маршрут | Содержимое |
|---------|------------|
| `/preferences` | **Оформление** (тема); блок «Сотрудники и роли» со ссылкой на хаб и (при праве) на «Админ-настройки»; внизу — **роль только для этой вкладки браузера** (локальная имитация до API, не путать с назначением ролей в справочнике пользователей). Список ролей в селекте — активные из справочника по `sortOrder`; если текущая сессия ссылается на неактивную роль, она показывается отдельной строкой, чтобы выпадающий список не ломался. |
| `/settings` | Матрица прав (без колонки суперадмина) + витрина правил полей (`field-rules.catalog.ts`). |
| `/dictionaries` | Плитки справочников; каждая может быть скрыта по `dict.hub.*`. Плитки «Роли» и «Пользователи» — CRUD по справочникам (см. [`users-dictionary.md`](./users-dictionary.md)). |

## Подключение API (будущее)

- Сервер отдаёт список ролей и для каждой роли (или для текущего пользователя) набор `PermissionKey`.
- `PermissionsService` получает метод вида `applyFromServer(...)`; локальная матрица в prod отключается или остаётся только для dev.
- Суперадмин на сервере: либо отдельный флаг, либо роль с неограниченным набором — фронт может по-прежнему не показывать её колонку в матрице организации.

## Ограничения и осознанные компромиссы

| Тема | Как сейчас | Риск / заметка |
|------|------------|----------------|
| Нет сервера | Права и роли в `localStorage` + mock-репозиторий | Любой с доступом к браузеру может менять роль сессии и матрицу; это **имитация**, не безопасность. |
| Один суперадмин | Один `isSystem` в seed | Не создавать второй `isSystem` без изменения правил в коде. |
| Код `admin` | Зарезервирован | Исключает коллизию с логикой полного доступа. |
| Роли без прав | По умолчанию `[]` | После обновления документа старые seed-роли «Редактор» / «Просмотр» не получают пресет из кода — их нужно один раз настроить в матрице (или выдать Excel-шаблон организации). |
| `PermissionsService` → `RolesStore` | Зависимость core → features | Прагматично до выноса контракта ролей в `core`; при росте проекта можно ввести токен «провайдер снимка ролей» только в core. |
