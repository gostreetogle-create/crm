# Справочник «Изделия»

**Продуктовое описание полей, пробелов и состава «изделие из изделий»:** [`docs/domain/products-guide.md`](../domain/products-guide.md).

**Не путать:** производственное изделие (эта плитка, API `/api/products`, модель `ManufacturedProduct`) — это не **каталожный товар** комплекса. Каталог «комплекс → товар → позиция»: [`catalog-suite-dictionary.md`](./catalog-suite-dictionary.md).

**Код:** плитка `products` на хабе `/dictionaries`, формы в `dictionaries-page-products.ts`, стор `ProductsStore`, API `GET|POST /api/products`, `PUT|DELETE /api/products/:id`.

---

## Не вижу плитку «Изделия»

Плитки хаба показываются только при праве **`dict.hub.products`** (маппинг в `dict-hub-permissions.ts`). Суперадмин видит всё. Если вы заходите под **директором / редактором** и плитки нет, почти всегда виновата **сохранённая матрица прав** в БД (`AppSetting` `authz_matrix`): её сохраняли до появления ключа `dict.hub.products`, и новый ключ в список не попал.

**Что сделать:** зайти под **admin** → **Админ-настройки** → матрица прав → для нужной роли включить **«Плитка «Изделия»»** (или **сбросить колонку роли на дефолт**). После сохранения обновить страницу. При странном кэше: выйти из системы и войти снова (очистится локальный кэш матрицы).

## Список изделий пустой

Это **нормально для новой БД:** в `prisma/seed.ts` **нет** демо-изделий — записи появляются только после **«Добавить изделие»** (+ у строки хаба или в панели таблицы). Если при загрузке с API ошибка (например, не применены миграции с таблицей `Product`), над таблицей показывается **красное сообщение**; проверьте в DevTools запрос **`GET /api/products`** и выполните **`npx prisma migrate deploy`** в каталоге `backend/`.

**Шапка изделия (поля API):** `code` (артикул, опционально, уникален если задан), `name`, `description` (витрина / КП), `priceRub`, `costRub`, `notes` (внутренние заметки), `isActive`. В развёрнутой таблице хаба — колонки «Артикул», «Описание»; в свёрнутой строке — `code — name` или только `name`.

**Состав изделия:** одна или несколько строк (`ProductLine`) — деталь производства (`productionDetailId`), опционально переопределение вида работ (`workTypeId`) и цвета (`colorId`). Цена/себестоимость на шапке изделия; при пустых значениях на бэкенде подставляется сумма итогов по строкам деталей (как в форме).

## Массовое наполнение (JSON / Admin API)

Единый Excel-импорт и кнопки Excel в `CrudLayout` **сняты**. Массовое создание — через **админские JSON endpoints** (см. `docs/dev-bulk-json-migration-checklist.md`). Пилот: `POST /api/admin/bulk/units`. Для изделий отдельный bulk-route появится при переносе доменной логики с прежнего Excel-потока.

**Ориентир полей** для будущего JSON (одна строка payload ≈ одна строка состава, несколько строк с одним изделием объединяются так же по смыслу):

| Поле | Смысл |
|--------|--------|
| ID изделия | UUID существующего изделия. Пусто — новое; заполнено — обновление (состав перезаписывается целиком). |
| Артикул | Необязательно; уникален среди непустых при импорте. |
| Наименование изделия | Обязательно. |
| Описание (витрина) | Необязательно; для клиента и КП. |
| Цена ₽, Себестоимость ₽ | Числа; пусто — подстановка суммы по итогам выбранных деталей. |
| Заметки | Внутренний текст. |
| Активен | да / нет. |
| Порядок | Порядок строки в составе (необязательно). |
| ID детали | UUID «Детали производства». Обязательно. |
| ID вида работ | Опционально. |
| ID цвета | Опционально. |

После импорта через API список на плитке обновляется обычным `getItems()` / перезагрузкой стора.
