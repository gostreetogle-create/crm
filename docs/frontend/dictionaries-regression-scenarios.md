# Регрессионные сценарии справочников (бэклог #12–14)

Краткие сценарии для ручной проверки перед релизом. Автотесты не заменяют эти цепочки.

## #12 — Геометрия и единица из модалки материала

**Цель:** из формы материала (модалка на хабе или полноэкранный «Новый материал») открыть создание геометрии / единицы, сохранить, убедиться что id подставился в поле материала.

1. Открыть **Материалы** → создать или редактировать материал.
2. У поля **Единица измерения** нажать «+» (или эквивалент быстрого создания) → заполнить минимально → сохранить.
3. Ожидание: модалка единицы закрылась, в поле материала выбрана новая единица.
4. Аналогично для **геометрии** (поле геометрии → быстрое создание → сохранить → подстановка id).

После проверки: **Назад** без сохранения материала — не должно залипать `unitQuickAddForMaterials` / `geometriesModalStackAboveMaterials` (см. сброс в `navigateBackFromNewMaterialPage` и закрытие вложенных модалок).

## #13 — Цепочка материал ↔ характеристика

**Цель:** сценарий «новый материал» → «новая характеристика» → сохранение характеристики → возврат → подстановка id в материал (через `DictionariesMaterialStandaloneFlowService`).

1. `/справочники/новый-материал` → открыть переход на новую характеристику (как в продукте).
2. Создать характеристику, сохранить.
3. Ожидание: возврат к форме материала, в поле характеристики подставлен новый id (после polling store, см. playbook).

Дополнительно: **Назад** с формы характеристики без сохранения — `cancelFlow()` и отсутствие «залипшего» pending id.

## #14 — Цвет / отделка / покрытие: propagation

**Канон:** модалки «данные связаны» (`LinkedDictionaryPropagationConfirm`) для **обновления** и **удаления** цвета, отделки, покрытия — глобальные оверлеи в шаблоне `dictionaries-page`; они не привязаны только к модалке хаба: тот же сценарий срабатывает при **редактировании** с полноэкранного маршрута standalone, если у записи есть связанные характеристики.

- **Создание** новой записи из quick-add в форме характеристики материала (`colorQuickAddForMaterialCharacteristics` и аналоги) — **без** propagation: прямое создание и подстановка id.
- **Редактирование** записи, на которую ссылаются характеристики — показ диалога local/global (или отмена), как на хабе.
- **Удаление** из таблицы с непустой связью — диалог delete propagation.

Итог для теста: убедиться, что propagation **не** показывается на голом create без связей, и показывается при edit/delete при `relatedCount > 0` в обоих контекстах (хаб и standalone), без дублирования логики API.
