# Video Docs Program

Формат: 4 обучающих видео для команды. Для каждого видео фиксируются: цели, сценарий, демо-шаги, частые ошибки, DoD.

## Видео 1 — Архитектура и границы

- Длительность: 20-30 минут.
- Цели:
  - объяснить архитектурные слои и границы ответственности;
  - показать, как CI ловит drift до merge;
  - научить читать fail-fast ошибки без "слепых" правок.
- Сценарий:
  1. Карта репозитория и роли `crm-web`/`srm-front`.
  2. Shared contracts и архитектурные guard-скрипты.
  3. Как устроен pipeline `frontend-ci`.
  4. Разбор реального failure-сценария.
- Демо-шаги:
  1. Запустить `npm run check:workspace-prereqs`.
  2. Запустить `npm run check:route-parity`.
  3. Показать, где находится и как читается `frontend-ci.yml`.
- Частые ошибки:
  - правки route только в одном приложении;
  - обход fail-fast вместо исправления причины;
  - локальные изменения без обновления docs.
- DoD:
  - инженер может перечислить обязательные архитектурные гейты;
  - инженер умеет диагностировать fail-fast и указать root cause;
  - инженер знает, какие файлы являются source of truth.

## Видео 2 — Словари и CRUD-контракты

- Длительность: 20-30 минут.
- Цели:
  - унифицировать подход к CRUD в хабе `/справочники`;
  - закрепить field contracts как обязательный процесс;
  - снизить риск регрессий form/table/payload.
- Сценарий:
  1. Канон маршрутов и redirects.
  2. Общий CRUD layout vs доменная логика.
  3. Field contract lifecycle.
  4. Разбор антипримеров и корректного решения.
- Демо-шаги:
  1. Изменить одно поле по канону (контракт -> форма -> payload -> таблица).
  2. Запустить `npm run check:dictionaries-page-size`.
  3. Проверить PR checklist блок B.
- Частые ошибки:
  - обновление только формы без payload/table;
  - локальные исключения без записи в deviations log;
  - дублирование CRUD-поведения в feature-слое.
- DoD:
  - инженер выполняет изменение поля без drift;
  - изменение подтверждается проверками и docs;
  - отсутствуют неучтенные временные отклонения.

## Видео 3 — PR-процесс и CI-гейты

- Длительность: 20-30 минут.
- Цели:
  - стандартизировать "definition of ready to merge";
  - исключить субъективные критерии в ревью;
  - зафиксировать единый формат отчета по изменениям.
- Сценарий:
  1. Прохождение `docs/frontend/pr-checklist.md`.
  2. Локальный pre-PR запуск проверок.
  3. Интерпретация CI-результатов.
  4. Шаблон финального отчета.
- Демо-шаги:
  1. Прогнать обязательные npm-check scripts.
  2. Показать, как диагностировать `No tasks were run` vs реальный PASS.
  3. Сформировать короткий risk register к PR.
- Частые ошибки:
  - merge без проверки docs;
  - смешение unrelated изменений в один PR;
  - игнорирование `Blocked` статусов.
- DoD:
  - инженер может самостоятельно провести PR по чеклисту;
  - отчет содержит проверки, риски и границы применимости;
  - CI-статусы интерпретируются корректно.

## Видео 4 — Поддержка канона и техдолг

- Длительность: 20-30 минут.
- Цели:
  - показать регулярный процесс поддержания архитектурного канона;
  - научить закрывать техдолг измеримо и без дрейфа;
  - закрепить обязательность `temporary-deviations-log`.
- Сценарий:
  1. Обзор канон-доков и их иерархии.
  2. Работа с deviations log.
  3. Планирование декомпозиции mega-files.
  4. Ежемесячный mini-audit и метрики.
- Демо-шаги:
  1. Смоделировать временное отклонение и корректную запись в log.
  2. Составить план закрытия долга с критериями.
  3. Привязать закрытие к PR/коммиту.
- Частые ошибки:
  - "временное" без срока закрытия;
  - обновление кода без синхронизации канон-доков;
  - отсутствие критериев завершения техдолга.
- DoD:
  - у команды есть повторяемый цикл mini-audit;
  - каждое отклонение имеет срок и владельца;
  - канон-доки остаются синхронны с практикой.
