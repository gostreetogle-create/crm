# Справочники: единый паттерн полей и заполнения

Цель: при добавлении любого нового справочника в `/dictionaries` — **одинаковое** понимание «что рисуем, как подписываем, как ведёт себя форма в разных режимах», без дрейфа названий и без дублирования смысла в пяти местах.

Связанные документы:

- Подписи длинные/короткие: [`dictionaries-naming-convention.md`](./dictionaries-naming-convention.md)
- Каркас CRUD и Excel: [`dictionaries-crud-playbook.md`](./dictionaries-crud-playbook.md)
- Чеклист задачи: [`new-dictionary-checklist.md`](./new-dictionary-checklist.md)
- Пример таблицы «домен ↔ колонка ↔ модалка ↔ Excel»: [`work-types-crud.md`](./work-types-crud.md)
- Откуда брать цены для себестоимости по справочникам: [`costing-and-dictionary-prices.md`](./costing-and-dictionary-prices.md)

## 1. Три слоя — не смешивать

| Слой | Что это | Правило |
|------|---------|--------|
| **Данные** | Имена в TypeScript, ключи `TableColumn.key`, поля DTO/API | Меняются только осознанно, при смене контракта. Не переименовывать ради перевода. |
| **Таблица** | `TableColumn.label` в `CrudLayout` | Короткие подписи из naming convention (плотная сетка, скан глазами). |
| **Модалка** | `label` у `UiFormField` внутри `UiModal` | **Полные** подписи того же поля (create / edit / **view** — одна и та же форма, в view поля только `disable`). |

Итог: одно поле = один стабильный `key`, но **две** русские строки (короткая в таблице, длинная в модалке), если для термина заведена пара «длинное / короткое».

### Сетка полей в модалке (`formGrid` / `formGridWide`)

На `/dictionaries` (и по аналогии в других CRUD):

- **`formGrid` только** — если **не больше шести** основных полей (пары label+control), не считая один общий блок «Заметки» на всю ширину.
- **+ `formGridWide`** — если основных полей **больше шести**: две колонки; у полей с классом `formWide` / `checkbox formWide` — `grid-column: 1 / -1`.

Порог раньше ориентировался на 4–5 полей; действующее правило — **>6 → две колонки** (см. `dictionaries-page.scss`).

### Секции в модалке (`fieldset.dictionaryFormSection`)

- Одна и та же оболочка для всех справочников на `/dictionaries`: `fieldset` + `legend` + тело `div.dictionaryFormSectionBody` (вариант `--2` для двух колонок). Длинные поясняющие абзацы под секциями не используем — только подписи полей и `legend`.
- Порядок секций — по логике заполнения: идентификация / учёт → ссылки на другие справочники → авто-поля (если есть) → коммерция / паспорт → заметки → статус.
- Блок **авто-полей** (значения подставляются из выбора выше): класс `dictionaryFormSection--autoFields` на `fieldset`. **Не использовать** `input readonly` / `UiFormField` с полями ввода — это выглядит как редактирование. Канон: сетка `dictionaryReadonlyGrid` внутри `dictionaryFormSectionBody`, строки `dictionaryReadonlyPair` → `dictionaryReadonlyLabel` + `dictionaryReadonlyValue` (при необходимости `dictionaryReadonlyValueRow` + `colorSwatch`). Значения по-прежнему можно держать в `FormControl` и обновлять `patchValue` — в шаблоне выводить `{{ form.controls.xxx.value }}`, без привязки к инпуту. Стили: `dictionaries-page.scss`.
- Геометрия: блок размеров остаётся `fieldset.geometryParams` («Параметры, мм») — тот же визуальный язык, что у `dictionaryFormSection`.

## 2. Поведение по сценариям

### Создание

- Обязательные поля — валидаторы и понятные сообщения под полем (на русском).
- Необязательные — без ложного `required`; default у булевых (например «Активна») задаём явно в коде и в UX-ожидании («новая запись по умолчанию активна»).
- После неуспешной отправки — `markAllAsTouched` / флаг попытки из store, как в существующих карточках.

### Редактирование

- Тот же набор полей и **те же полные подписи**, что при создании.
- Заголовок модалки отражает режим («Редактирование …»), формат согласовать с playbook.

### Просмотр (View)

- **Та же разметка полей**, что у edit, но форма `disable`, в футере — «Закрыть», без «Сохранить».
- Заголовок: `Просмотр <длинное название сущности> — <короткий тег>` (playbook).

### Дублирование (Duplicate)

- Новая запись в режиме создания: переносим значения так, чтобы не нарушать уникальность **бизнес-ключа** (код/обозначение): типично суффикс `-КОПИЯ` к коду и/или «(копия)» в наименовании — правило фиксируем в feature-doc конкретного справочника.
- Пользователь перед сохранением правит код/имя при необходимости.

### Удаление

- Подтверждение через `UiModal` в `CrudLayout`, не `window.confirm`.
- Текст подтверждения можно уточнить под сущность («Удалить вид работ?»).

## 3. Placeholder (`placeholder` в input)

Назначение: **пример допустимого значения** или короткий образец формата (`Напр. Св.`), а не длинная инструкция.

Если пользователи часто путают поле с **другой сущностью** (например «сюда не вносят нормы времени»):

- либо **одна короткая фраза** в placeholder (если команда согласна держать правило в UI),
- либо **только** в [`справочники/crm-manufacturing-dictionaries.json`](../../справочники/crm-manufacturing-dictionaries.json) (`description` / `domainVsUi`) + в feature-doc справочника — без «простыни» в каждом поле.

Не дублировать одно и то же пояснение и в placeholder, и в `errorText`, и в тултипе, и в карточке — достаточно **одного** выбранного места + ссылки в doc.

## 4. Excel (шаблон / импорт / экспорт)

- В файл попадают **только бизнес-колонки**; системные поля (`id`, `row_version`, …) не в шаблоне до появления upsert.
- Заголовки листа = те же «человеческие» имена, что в шаблоне и в валидаторе импорта (один источник правды в коде).
- Импорт в текущем каноне — **только создание** новых строк (`createMany`); ошибки — с номером строки и причиной, без «тихих» пропусков.
- Нормализация текста, сопоставление Excel с мелкими справочниками без id в файле, снимки на материале и **чеклист аудита таблиц** — [`dictionaries-data-and-import-rules.md`](./dictionaries-data-and-import-rules.md).

## 5. Как вводить новый справочник без хаоса

1. Запись в manufacturing JSON (поля, описание, статус).
2. Короткий **feature-doc** со **таблицей**: доменное имя | `TS`/key | `label` колонки | `label` модалки | колонки Excel.
3. При необходимости — новая строка в `dictionaries-naming-convention.md` (пара длинное/короткое для **нового** термина).
4. Реализация по [`new-dictionary-checklist.md`](./new-dictionary-checklist.md).
5. Проверка: колонка короткая, модалка полная, сценарии create/edit/view/duplicate/delete согласны с этим документом.

## 6. Анти-паттерны

- Менять `key` колонки, чтобы «поправить русское слово» — нельзя; меняем только `label`.
- Разные сокращения для одного термина в двух таблицах без записи в naming convention.
- В модалке сокращения как в таблице без веской причины — ломает режим View для новых пользователей.
- Длинные юридические тексты в placeholder — вынести в doc или подсказку одного уровня.

Этого достаточно, чтобы следующие справочники делались **однотипно** по полям и по поведению; детали домена — в JSON и в feature-doc конкретной сущности.
