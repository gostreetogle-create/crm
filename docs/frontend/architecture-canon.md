# Frontend Architecture Canon

## 1. Source of truth

- Route contracts, shared CRUD behavior и публичные redirect-paths задаются в `libs`.
- Приложения (`crm-web`, `srm-front`) только подключают shared контракты и app-specific страницы.

## 2. Mandatory quality gates

- Fail-fast:
  - `check:workspace-prereqs`
  - `check:route-parity`
  - `check:dictionaries-page-size`
- Основной CI:
  - `nx affected -t lint`
  - `nx affected -t test`
  - `nx affected -t build`

## 3. Dictionaries architecture

- Единый пользовательский вход: `/справочники`.
- Legacy redirects поддерживаются, но не считаются отдельными UI-потоками.
- CRUD-поведение (toolbar/rows/actions) унифицируется через shared layout.

## 4. Field contract canon

- Поле считается каноничным только если синхронизировано в:
  - форме,
  - payload builder,
  - таблице,
  - тесте/проверке.
- Поля без typed-контракта считаются техническим долгом.

## 5. Mega-file policy

- Файлы > 3000 строк считаются зоной повышенного риска.
- Для `dictionaries-page.ts` действует временный guard по размеру (6200 строк).
- Любая крупная фича должна выносить новую доменную логику в отдельные модули/helper-файлы.

## 6. Deviation policy

- Временные исключения допускаются только через `temporary-deviations-log.md`.
- Исключение без записи = нарушение канона.

## 8. Правила безопасной работы с ИИ-ассистентом

- Критичные зоны (любые правки — только после явного согласования):
  - `system-status.store` и любые файлы вида `**/state/*.store.ts` (signal stores, глобальное и фичевое состояние).
  - Auth: вход, сессия, guards, клиенты и обработчики вокруг `/api/auth/*`.
  - RBAC: роли, проверки прав, админ-настройки и всё, что определяет «кто что видит/может».
  - Статусы и жизненный цикл КП (коммерческие предложения): переходы, отображение, согласование с бэкендом.
- Согласование перед правкой:
  - Перед любым изменением в перечисленных зонах ИИ обязан сначала явно написать фразу по шаблону: «Это затронет … Риск сломать …. Делаем?» — с конкретным текстом вместо многоточий (что меняется и что может отвалиться у пользователей или в CI).
  - Правки в эти зоны молча, без этого сообщения и без явного «да» от разработчика, запрещены.
- Пост-пулловый ритуал (после `git pull`, особенно если менялись зависимости, Prisma-схема или примеры конфигурации):
  - `npm install` в затронутых пакетах (как минимум `crm-web/`; при работе с API — `backend/`).
  - В `backend/`: `prisma generate` и при необходимости применение миграций (`migrate` / актуальная для команды команда деплоя схемы).
  - Проверка `backend/.env`: обязательные переменные на месте, в т.ч. `DATABASE_URL` и прочие ключи, без которых падает локальный запуск.

### Правило завершённости больших изменений

Если задача требует изменений в нескольких файлах или затрагивает архитектуру, ИИ обязан:

- Довести изменение до полностью рабочего состояния (сборка проходит, ошибок нет)
- **Либо** создать отдельный чек-лист в `docs/todo/` с названием задачи и всеми пунктами, которые остались доделать
- После большого изменения всегда писать короткий статус: что сделано, что осталось, есть ли ошибки сборки

Не оставлять «полуготовые» изменения.

## 9. Будущее развитие

- Постепенный переход к модульной архитектуре (целевое правило, внедрение без ломки текущего кода):
  - Каждый крупный блок (Админка, КП, Справочники, Авторизация) должен жить в своём feature-lib.
  - Модули общаются только через публичный API (экспорты библиотеки), а не через «проходные» файлы соседа.
  - Запрещено напрямую импортировать внутренности другого модуля (пути вне публичного surface).
  - Правило вводится постепенно, по одному модулю за раз — без требования переписать всё сразу.
