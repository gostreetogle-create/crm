# Development workflow (обязательный)

Цель: все следующие задачи делаются в одном качестве и в одном формате, как текущие правки.
Статус правила: всегда обязательно для всех следующих изменений в проекте.

## Порядок чеклистов (что проходить первым)

1. **Глобальная проверка сборки и загрузки** (запланированный день / перед крупным релизом / после долгой серии фич): целиком [`performance-and-build-checklist.md`](./performance-and-build-checklist.md) — замеры `nx build` и Docker, размер бандла, lazy/deps, чистка. **Без** SSR, Universal и других новых технологий; только оптимизация текущего стека.
2. **Крупные UI-изменения перед merge:** по затронутым разделам [`visual-consistency-checklist.md`](./visual-consistency-checklist.md).
3. **Перед сдачей каждой задачи / PR:** [`pr-checklist.md`](./pr-checklist.md).

Почему наследие в коде не исчезает «само» после обновления доков, почему массовое выравнивание делается поэтапно, и что делать, если в каноне не хватает детали: [`ui-consistency-and-legacy.md`](./ui-consistency-and-legacy.md).

Для полного AI-аудита и пошагового внедрения UI/UX (формат для нетехнического владельца): [`ai-ui-audit-agent-prompt.ru.md`](./ai-ui-audit-agent-prompt.ru.md).

**Локальный запуск (Postgres + backend + `nx serve`):** [`backend-enable-runbook.md`](./backend-enable-runbook.md); для ассистента в Cursor — `.cursor/rules/local-dev-launch.mdc`.

**Для задач по КП и сложной БЛ статусов:** использовать целевой roadmap [`commercial-offers-bl-roadmap.md`](./commercial-offers-bl-roadmap.md) как обязательный ориентир по порядку внедрения.

**Минимальный release gate (critical tests):** [`../release-gates.md`](../release-gates.md).

**Короткий спринт до сдачи (~6 дней):** чеклисты MVP, smoke и вечерние gate — [`../dev-sprint-6d-runbook.md`](../dev-sprint-6d-runbook.md).

## Прод-деплой на VPS (фронт)

**Один только `git push` новый UI на сервер не доставляет:** статика не в git, она лежит на диске VPS в **`deploy/prebuilt-web/`**.

Каждый раз, когда нужно обновить интерфейс на сервере:

1. Локально — **сразу после push** (или перед ним) — production-сборка: `cd crm-web` → `node scripts/sync-canonical-roles.cjs` → `npx nx run crm-web:build:production`.
2. Выкладка статики (отдельно от git): **`deploy/prebuilt-web.zip`** в **`deploy/`** на сервере (собрать: `deploy/scripts/pack-prebuilt-web.ps1` / `pack-prebuilt-web.sh`) **или** содержимое `browser/` в **`deploy/prebuilt-web/`** (rsync/scp/SFTP).
3. На сервере — **`./deploy.sh`** в `deploy/`.

Таблица шагов и примеры (включая Windows): **[`deploy/README.md`](../../deploy/README.md)**. Для ассистента в Cursor при обсуждении деплоя — `.cursor/rules/deploy-prebuilt-web.mdc`.

## Стандарт выполнения каждой задачи

1. Сначала обновляем/создаём контракт (тип/интерфейс/модель), потом UI и только потом детали.
2. Используем существующие компоненты из **`@srm/ui-kit`** (`crm-web/libs/ui-kit/`); если повторение >= 2 раз, выносим примитив в **`libs/ui-kit`**.
   - Если нужного компонента нет: сначала добавляем его в `crm-web/libs/ui-kit/src/lib/...`, экспортируем из `libs/ui-kit/src/index.ts`, затем импортируем в фичах как `@srm/ui-kit`.
3. Любые стили в фичах строим через theme tokens (`var(--...)`), без локальных палитр.
   - Для типовых UI-иконок используем `@lucide/angular` как базовую библиотеку.
   - Цвет иконок только через токены темы (`--icon-affirm`, `--accent`, `--warning`, `--danger`, `--text-muted`), без hardcoded hex в feature-коде.
   - Если добавлен новый цветовой токен иконок, его нужно сразу провести через весь поток темы: `crm-web/libs/theme-core/src/lib/theme-schema.ts` → `theme-presets.ts` → runtime в `theme.store.ts`.
4. Для форм:
   - используем Reactive Forms как стандарт (template-driven/`ngModel` не использовать в новых и поддерживаемых формах),
   - базовая валидация обязательна сразу (required/min/minLength и т.д.),
   - ошибки показываются рядом с полем,
   - тексты ошибок и UI-подписи — на русском и в понятной форме.
   - обязательное поле в ошибке визуально подсвечивается красной рамкой (из общего токена), даже если текст под полем отключён.
  - если редактирование переносится в диалог, использовать полноценный UI-модал (не `alert/prompt`);
  - закрытие диалогов:
    - `View`: `Esc` + backdrop + явная кнопка,
    - `Create/Edit`: `Esc` + явная кнопка, без закрытия по backdrop.
   - для единообразия использовать shared-компонент поля (`UiFormFieldComponent`) вместо локальной разметки `label + input + error`.
5. Для справочников и CRUD:
   - список + создать + изменить + удалить,
   - mock-репозиторий как минимум до готовности backend,
   - **пользовательский UI справочников — только хаб `/dictionaries`** (редиректы со старых путей не считаются отдельным экраном); ориентир порядка: `docs/frontend/dictionaries-crud-playbook.md` (раздел «Порядок: единый хаб»),
   - роут приложения для пользователя — `/dictionaries`; DI (репозиторий + store) подключается в `app.routes.ts` внутри этого route в той же задаче,
   - ориентир по реализации: `docs/frontend/dictionaries-crud-playbook.md`,
   - RBAC (ключи прав, матрица, роли): **`docs/frontend/authz-roles-architecture.md`**, операции и диагностика — `docs/frontend/authz-matrix-runbook.md`; устаревший обзор: `docs/frontend/rbac-and-admin-settings.md`. Справочник пользователей на хабе: `docs/frontend/users-dictionary.md`.
   - запрещены точечные правки "только в одном месте": правка паттерна обязана пройти по всем связанным карточкам/экранам в том же изменении.
6. После изменений обязательно:
   - `nx build crm-web`,
   - проверка линтера для затронутых файлов,
   - обновление docs по изменённой фиче.
   - если есть временное локальное отклонение — добавить запись в `docs/frontend/temporary-deviations-log.md`.

## Как понимать формулировки про «карточку на хабе»

На `/dictionaries` все плитки справочников — **один и тот же шаблон** (`CrudLayout` внутри `app-dictionary-hub-tile` → `app-content-card`; при необходимости ширины на всю сетку — тот же тег с `[fullWidth]="true"`). Различаются только данные, колонки и модалки домена.

- **По умолчанию:** фразы вроде «на карточке цветов», «в справочнике материалов кнопка…», «в плитке покрытий поле…» применительно к **таблице/тулбару/кнопкам строки** означают правку **общего** места: `crm-web/libs/ui-kit/src/lib/crud-layout/` (импорт **`CrudLayoutComponent`** из `@srm/ui-kit`) и при необходимости токены темы, **а не** только `colors` / `materials` и т.д. Иначе архитектура расходится: одна плитка выглядит иначе, остальные — нет.
- **Исключение:** правка **только одной** плитки (нарушение единообразия намеренно) — только если в задаче **явно** сказано, например: «только карточка цветов», «исключение для одной плитки», «остальные не трогать». Такое исключение обязательно зафиксировать в `docs/frontend/temporary-deviations-log.md` с причиной и планом выравнивания.
- **Домен без общего шаблона:** если речь о **содержимом модалки** или **полях формы** конкретного справочника — правки в соответствующем блоке хаба / форме (это не ломает единый каркас плитки).

Это правило снимает типичную ошибку: пользователь называет справочник как **пример того, где он видит баг**, а имел в виду **общий паттерн всех карточек**.

## Запреты (чтобы не уехать от стандарта)

- Нельзя добавлять новый UI-паттерн в feature-код, если его нет в **`@srm/ui-kit`**:
  сначала создаём компонент в `libs/ui-kit`, потом используем.
- Нельзя оставлять конфликт в документации:
  при расхождении правим docs в том же изменении, без откладывания.
- Нельзя закрывать задачу с частичной синхронизацией UI-паттерна:
  если меняется `CrudLayout`/карточки/роли, изменение обязательно применяется во всех связанных экранах (`/demo`, `/dictionaries`, актуальные справочники).
- Если полная синхронизация сейчас технически невозможна:
  оформить "чеклист на будущее" в docs (список несинхронизированных мест + критерий завершения + срок).
- Нельзя оставлять временное локальное решение без записи в журнале отклонений:
  `docs/frontend/temporary-deviations-log.md`.
- Нельзя выпускать форму без единого UX-правила ошибок
  (обязательное поле, момент показа ошибки, текст ошибки на русском).

## Что обновлять в docs после каждой задачи

- Док фичи (`docs/frontend/<feature>.md`) — что сделано, какие поля/правила.
- Если изменились общие правила — `design-system.md` и/или `ui-reuse.md`.
- Если менялась архитектура — `feature-structure.md`.

## Definition of Done (DoD)

Задача считается законченной только когда:

- код собран (`nx build crm-web` успешно),
- нет новых lint-ошибок в изменённых файлах,
- UI работает и соответствует текущему style contract,
- документация синхронизирована с кодом.
- для затронутого паттерна нет несогласованных копий в других карточках/экранах (или есть явный открытый checklist-долг).

## Перед сдачей задачи

Обязательно пройти чеклист: `docs/frontend/pr-checklist.md`.  
Если в спринте запланирован **день глобальной проверки** — сначала `docs/frontend/performance-and-build-checklist.md`, затем PR-чеклист и остальное по списку выше.
