# Roadmap: КП, статусы и BL-устойчивость

Цель: зафиксировать единый план эволюции логики коммерческих предложений (КП), чтобы при росте процессов не появлялись регрессии и "плавающее" поведение UI.

## Почему это нужно

- Логика статусов и связанных действий (например, создание заказа при оплате) стала критичной.
- Ошибки класса "состояние одной строки влияет на другую" возникают, когда source of truth размыт между шаблоном, стором и API.
- Нужны единые доменные правила и предсказуемое поведение фронта/бэка.

## Архитектурный стандарт (принято)

- Frontend state: стандарт на `NgRx SignalStore` для сложных сущностей и табличных сценариев.
- Статусы КП: единый модуль правил переходов (нормализация + guards + transition-map).
- Backend: бизнес-логика в сервисах (application layer), роуты только transport/validation.
- Инварианты домена: обеспечиваются в БД/транзакции и дублируются защитой на уровне API.
- Ошибки: стабильные machine-readable коды + единый маппинг на пользовательские сообщения.

## План внедрения

### Этап 1. Стабилизация фронта статусов КП

1. Вынести правила статусов в общий модуль (`normalize + canTransition + transition map`).
2. Убрать дубли правил из компонентов/страниц.
3. Проверять доступность перехода только по актуальному статусу из store по `id`.
4. Закрепить per-row lock во время запроса смены статуса.

Статус: **в работе** (первый шаг уже внедрен).

### Этап 2. Backend service layer для статусов

1. Выделить `changeOfferStatus` в сервис.
2. Выделить `createOrderFromPaidOffer` в сервис.
3. Оставить в роуте только валидацию входа и маппинг ошибок.
4. Явные доменные ошибки: `illegal_status_transition`, `paid_offer_locked`, `offer_has_order`, `invalid_current_status`.

Статус: **завершен** (выделены `changeOfferStatus` и `createOrderFromPaidOffer`, роут статусов переведен на сервис + централизованные коды ошибок).

### Этап 3. Тесты критического контура

1. Backend integration: `draft -> waiting`, `draft -> paid (409)`, `waiting -> paid`, повторные изменения.
2. Проверка инварианта: строго 1 заказ на 1 КП.
3. Frontend store tests: нормализация статусов, transition guards, обработка 409.

Статус: **завершен** (backend: `vitest` + unit/HTTP-контракт покрытие критичного контура; frontend: добавлен test-контур для `dictionaries-state` и unit-тесты правил статусов КП с нормализацией legacy-ключей и матрицей переходов).

### Регламент проверки (release gate)

Каноничный источник команд и stop/go-правил:

- `docs/release-gates.md`

Кратко для контура КП:

- backend critical: `cd backend && npm run test:critical`
- frontend critical: `cd crm-web && npm run test:critical`

Покрытые сценарии (must-pass):

- Доменные правила статусов: нормализация `proposal_approved`, допустимые и недопустимые переходы.
- Сервис смены статуса: `not_found`, `invalid_current_status`, `illegal_status_transition`, валидные переходы с/без создания заказа.
- Инвариант `1 заказ на 1 КП`: отсутствие дубля при повторном paid.
- HTTP-контракт `POST /commercial-offers/:id/status`: корректный маппинг `404/409/200`.
- HTTP-контракт `DELETE /commercial-offers/:id`: корректный маппинг `404/409/204`, включая `offer_has_order` и `paid_offer_locked`.
- Frontend rules (`dictionaries-state`): нормализация и матрица переходов в `commercial-offer-status.rules`.

Критерий прохождения этапа:

- Все тесты из `docs/release-gates.md` зеленые локально.
- Любое изменение БЛ статусов/заказов сопровождается обновлением тестов в том же PR.

## Definition of Done по этой инициативе

- Правила статусов определены в одном месте и используются повторно.
- Нет прямых переходных правил в шаблонах UI.
- Backend-роуты не содержат предметную оркестрацию.
- Есть автотесты на критичные переходы и инварианты.
- Сценарий "изменение статуса в строке A ломает строку B" не воспроизводится.

Текущая оценка:

- Этап 1: в работе.
- Этап 2: завершен.
- Этап 3: завершен.

## Следующий контур после Этапа 3

- Распространить тот же подход contract-first тестов на `orders` API.
- Уже добавлен и расширен backend HTTP-контракт тест для `orders` (`GET/POST/PUT/DELETE` + ветки `invalid_body`, `invalid_offer_ref`, `order_exists_for_offer`, `not_found`).
- На frontend в `dictionaries-state` добавлены unit-тесты маппинга ошибок `orders` (backend error code -> пользовательское сообщение) через pure helper `orders-error-mapping`.
- На frontend в `dictionaries-state` добавлены unit-тесты маппинга ошибок `commercial-offers` (status/delete) через pure helper `commercial-offers-error-mapping`.
- В e2e smoke (`crm-web/e2e/srm-front-e2e-smoke.spec.ts`) добавлен сценарий UI-потока статусов КП до создания заказа.

## Техническая отметка о текущем старте

В рамках старта внедрен общий модуль правил статусов в `dictionaries-state`, и страница `/dictionaries` переведена на использование этих правил из единой точки. На backend выделены сервисы смены статуса/создания заказа, а правила статусов вынесены в общий `lib`-модуль для повторного использования роутом и сервисами.
