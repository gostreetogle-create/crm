# Матрица прав (роли ↔ разрешения)

Канон модели: **`docs/frontend/authz-roles-architecture.md`** (источники истины, чеклист разработчика).

Операционный чеклист после изменений ролей или жалоб «галочка есть — раздел не открывается».

## 0. Чистая БД (с нуля)

Канонические роли и пользователи задаются сидом: `backend/prisma/seed-roles.ts`, `backend/prisma/seed.ts`. Старая БД не нужна — проще пересоздать:

1. Убедиться, что PostgreSQL доступен, в `backend/.env` задан `DATABASE_URL`.
2. Из каталога `backend/`: `npm run db:reset` (миграции + seed). Либо вручную: удалить БД/volume и снова `prisma migrate dev` + `db seed`.
3. Локально по умолчанию создаётся тестовый вход **director / director** (роль «Директор»). Отключить: `SEED_DIRECTOR_USER=0` в `.env`. В Docker backend образе это уже `0`.

## 1. Данные в БД

- Таблица `Role`: у канонических ролей стабильные `id` (`role-sys-admin`, `role-seed-director`, … — см. сид `backend/prisma/seed.ts` и `crm-web/libs/roles-data-access`).
- Таблица `User.roleId` должна совпадать с `Role.id` пользователя; колонка матрицы «Директор» действует только на пользователей с этим `roleId`.

## 2. Серверная матрица

- `GET /api/authz-matrix` — значение из `AppSetting` (`authz_matrix`), затем **санитизация** как при записи: только известные `roleId` из таблицы `Role`, только валидные ключи, без `dict.hub.*` если нет `page.dictionaries`. Пустой объект после очистки → в ответе `matrix: null`. Код: `backend/src/lib/authz-matrix-sanitize.ts`, маршрут `backend/src/routes/authz-matrix.routes.ts`.
- `PUT` применяет ту же санитизацию к телу (неизвестные роли и неверные ключи отбрасываются; канон ключей — `backend/src/lib/authz-permission-keys.ts`).

## 3. Диагностика

- **CLI** (из каталога `backend/`): `npm run authz:check` — печать JSON и код выхода `1`, если в матрице есть «висячие» роли или неверные ключи.
- **API** (только админ): `GET /api/authz-matrix/diagnostics` — то же в виде отчёта для UI (кнопка на странице «Админ-настройки»).
- **Система и БД** (только админ): `GET /api/admin/system/status` — доступность PostgreSQL, сверка миграций с образом, готовые команды и промпты для ИИ; в UI — карточка «Система и обслуживание» на той же странице. Разбор логов и `requestId`: `docs/dev-logs-and-diagnostics.md`.

## 4. Клиент

- После успешного **`/auth/me`** (и после логина): **`RolesStore.ensureRolesLoaded()`** → **`PermissionsService.syncMatrixFromServerSafe()`**. Без загрузки ролей prune матрицы и таблица админа могли расходиться с БД.
- **`syncMatrixFromServerSafe()`** внутри вызывает полный синк с сервера, но **пропускает** запрос, если висит debounced сохранение матрицы на сервер (чтобы не затереть несохранённые галочки админа).
- Ошибка `GET` (сеть, 5xx) отражается в **`matrixSyncError`**; полоса в шапке, сброс вручную. Периодический опрос матрицы для пользователей без `page.admin.settings` (~120 с) и синк при возврате на вкладку (`visibilitychange`) используют тот же safe-метод.
- Код роли в формах API приводится к **нижнему регистру** (`POST/PUT /api/roles`), на фронте при расчёте дефолтов используется `code.toLowerCase()`.

## 5. Поведение матрицы

- Пока для `roleId` **нет** сохранённого переопределения — действуют дефолты из `DEFAULT_ROLE_PERMISSIONS_BY_CODE` (`crm-web/libs/authz-core/src/lib/authz.matrix.ts`): для **`editor` / `director` / `accountant`** — полный рабочий набор без раздела «Админ-настройки» (`DEFAULT_PERMISSIONS_BUSINESS_WORKSPACE`); для **`viewer`** — хаб справочников; для неизвестного кода — `[]`.
- После первого сохранения набора галочек для роли — **только этот список** (полная замена дефолта). Кнопка **«Сбросить галочки»** в колонке сохраняет для роли **явный пустой список** (`[]` в матрице): все галочки сняты, прав нет (это не «вернуть дефолт по коду»).

## 6. Регрессия

- Проверять под **реальным** пользователем с нужной ролью, не только под админом (у суперадмина матрица не ограничивает).
