# Interaction system standard (Foundation v1)

Цель: обеспечить одинаковое и предсказуемое поведение UI на всех страницах CRM.

Визуальное единообразие (отступы, типографика, контролы, состояния, адаптив): [`visual-consistency-checklist.md`](./visual-consistency-checklist.md).

## Принцип "один эталон -> переиспользование везде"

1. Сначала доводим до эталона одну demo-страницу.
2. Эталонная страница собирается только из компонентов **`@srm/ui-kit`** (`crm-web/libs/ui-kit/`).
3. Любая новая feature-страница повторяет поведение и визуальный паттерн эталона.
4. Изменение базового поведения вносится в **`libs/ui-kit`** и автоматически отражается везде, где импортированы те же примитивы.

## Обязательные UX-паттерны

1. `Edit` всегда открывает одинаковый сценарий редактирования (модалка/панель по выбранному стандарту).
2. `Create` всегда открывает одинаковый сценарий создания и одинаковый набор действий (`Сохранить`, `Отмена`/`Сбросить`).
3. Кнопка `+` рядом с полем связи открывает `quick-add` диалог:
   - создаем/выбираем сущность;
   - выбранное значение сразу подставляется в текущую форму.
4. Закрытие диалога всегда работает одинаково:
   - для `View`: `Esc`, клик по backdrop, явная кнопка закрытия;
   - для `Create/Edit`: явная кнопка закрытия (`Отмена`/`Закрыть`) и `Esc`;
   - для `Create/Edit` клик по backdrop не закрывает диалог (защита от случайной потери данных).
5. Поведение подтверждения удаления всегда одинаковое.
6. Нельзя использовать нативные браузерные окна (`window.alert/confirm/prompt`) для рабочих сценариев CRM.
   - Подтверждения/предупреждения делаем только через переиспользуемый `UiModal` в стиле приложения.

## Demo page как source of truth

Должна существовать отдельная demo-страница (доступ через header), где собраны:

- типы кнопок;
- поля и валидация;
- таблица и row-actions;
- модалки;
- quick-add/select-and-insert сценарий;
- состояния (`loading`, `empty`, `error`, `success`).
- иконографика ключевых сценариев (create/save/cancel/confirm, filters, row-actions, pagination) с семантическими цветами.

Любая правка визуала/поведения сначала валидируется на demo-странице, затем используется в feature-экранах.

## Header/navigation baseline

1. В приложении должен быть единый header.
2. В header должна быть явная кнопка/пункт `Demo`.
3. Навигация между ключевыми разделами должна быть унифицирована.

## Правила изменений

1. Нельзя внедрять уникальное поведение только в одной feature-странице, если это общий сценарий.
2. Если поведение повторяется >=2 раз, оно должно быть вынесено в **`@srm/ui-kit`** + задокументировано.
3. Все UX-правила фиксируются в docs перед масштабированием на бизнес-логику.
4. Любая замена набора иконок или изменение семантики цветов сначала валидируется на demo-странице, затем переносится в feature-экраны.

## Почему это важно

- Пользователи (5-6 человек, работа почти одновременно) быстро привыкают к единой логике.
- Снижается стоимость обучения и количество ошибок.
- Ускоряется разработка: меняем в одном месте, получаем эффект во всех экранах.
