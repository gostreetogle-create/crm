# Dictionaries Naming Convention (RU, ГОСТ-style)

Цель: единый стандарт именования для всех текущих и будущих справочников в CRM.

## 1) Общий принцип

- У каждого справочника есть:
  - длинное наименование (для карточки/раздела),
  - короткое наименование (для заголовков полей и колонок).
- Длинные названия: понятные бизнес-пользователю.
- Короткие названия: краткие, однозначные, близкие по стилю к техдокументации/ГОСТ.
- Сокращения применяются только в UI-подписях, не в API-ключах и endpoint.

## 2) Каноничные пары (текущий список)

- `Единицы измерения` -> `Ед. изм.`
- `Тип отделки / шероховатость` -> `Тип отд.` / `Шерох.`
- `Форма и габаритные размеры` -> `Форма / габ. разм.` (если нужно сокращение в узких колонках)
- `Цвета (RAL)` -> `Название` (без сокращения для поля/колонки названия цвета)
- Короткий тег для вставок в другие справочники (в title модалки): `Цвет`

## 3) Правила сокращений (ближе к ГОСТ)

- Использовать точку в сокращениях: `Ед. изм.`, `Тип отд.`, `Шерох.`, `габ.`
- Не сокращать до неочевидных форм (`Тип отд` без точки, `Едизм`, `Шрх` и т.д.)
- Не смешивать 2+ варианта для одного термина в разных местах.
- Если термин спорный — сначала добавить пару "длинное/короткое" в этот документ, потом внедрять в UI.

## 4) Область применения

- Карточки справочников в `/dictionaries`.
- Заголовки таблиц (`CrudLayout` columns).
- Label полей в формах модалок.
- Заголовки модалок в режиме просмотра: формат `Просмотр <длинное> — <короткое>`.
- Tooltip/aria-label для кнопок — использовать длинный понятный вариант.

## 5) Не распространяется на

- API DTO и JSON-поля (`name`, `code`, `isActive`, ...).
- Названия endpoint (`/units`, `/surface-finishes`, ...).

## 6) Definition of Done для новых справочников

- Добавлена пара "длинное/короткое" в этот файл.
- Применено одинаково в карточке, колонках и полях формы.
- Проверено отсутствие альтернативных сокращений в проекте (`rg` по ключевым словам).
