# Чеклист: сборка, загрузка сайта, чистка (без новых технологий)

Репозиторий: **Angular + Nx** (`crm-web/`). Прод на VPS по умолчанию: **готовая статика** в `deploy/prebuilt-web/` + лёгкий nginx (`deploy/Dockerfile.web`); полная сборка SPA в Docker — `deploy/Dockerfile.web.in-docker-build`.  
Цель: ускорить **локальную/CI/Docker-сборку** и **первую отрисовку** в браузере, убрать лишнее — **без** внедрения SSR, Edge Functions, смены фреймворка и т.п. Только то, что можно сделать на текущем стеке.

Оценки в колонке **Эффект**: **+** заметный выигрыш, **~** умеренный / ситуативный, **−** риск регрессий или много ручной работы без гарантии.

## 1. Сборка Angular/Nx (почему «Building…» долго)

| Сделать | Зачем | Эффект |
|--------|--------|--------|
| На слабом VPS: **не** гонять `nx build` на сервере — выкладывать **`dist/crm-web/browser/`** в `deploy/prebuilt-web/` (см. `deploy/README.md`) | Стабильный деплой без OOM | + |
| Если всё же **полная сборка в Docker** (`Dockerfile.web.in-docker-build`): **`NX_DAEMON=false`**, verbose, **`NG_BUILD_MAX_WORKERS`**, **`NODE_BUILD_HEAP_MB`**, **swap** 2–4 ГБ | Меньше OOM и «тихого» exit 1 | + |
| **CI / registry**: собирать образ web там и на сервере только `pull` — *опционально, см. compose* | Без тяжёлой сборки на VPS | ~ |
| Проверить, не копируется ли в контекст сборки лишнее (корневой `.dockerignore` при необходимости: `**/node_modules`, `dist`, `.git`) | Меньше контекста → быстрее `docker build` | + |
| Разово сравнить время **`nx build crm-web` с кэшем Nx** и с **`NX_SKIP_NX_CACHE=true`** | Понять, даёт ли кэш на CI/сервере | ~ |
| Убедиться, что **`defaultConfiguration: production`** для `build` — одна понятная цель | Меньше путаницы с dev-бандлами в проде | ~ |
| В **CI** (GitHub Actions и т.п.): собирать только **affected** (`nx affected -t build --base=...`) если пайплайн это поддерживает | Меньше минут на PR без затрагивания crm-web | + |
| Проверить **`tsconfig`**: `incremental` / стабильный `outDir` для вспомогательных tsc-задач | Ускорение вспомогательных проверок | ~ |
| Отключить лишние **source maps в production** (в Angular production обычно уже off) | Меньше работы билдера и размер артефакта | ~ |

## 2. Размер бандла и загрузка (то, что уже есть в Angular)

| Сделать | Зачем | Эффект |
|--------|--------|--------|
| **`nx build crm-web` → отчёт по чанкам** (в логе уже есть размеры чанков) | Найти самые тяжёлые куски | + |
| **Lazy routes**: тяжёлые фичи только через `loadComponent` / `loadChildren` (где ещё не сделано) | Меньше JS на первом экране | + |
| Тяжёлые библиотеки (внешние парсеры, графики и т.п.): только lazy-import на нужном маршруте/действии, не в корневом бандле | Меньше initial JS | + |
| Проверить **импорты из `@srm/*`**: не тянуть ли целую библиотеку ради одного типа | Tree-shaking лучше при точечных импортах | ~ |
| **Картинки в `public/`**: сжать PNG/WebP, не класть огромные оригиналы без нужды | Меньше трафика на КП/брендинг | + |
| **Шрифты**: `@fontsource/*` — подключать только нужные начертания/подмножества | Меньше CSS/woff | ~ |
| **Иконки**: не импортировать весь пакет иконок, если достаточно подмножества Lucide | Меньше мёртвого кода | ~ |
| Проверить **`styles` в `project.json`**: нет ли неиспользуемых глобальных SCSS | Меньше CSS в initial | ~ |
| Включить **`budgets`** в `angular.json` / `project.json` для `crm-web` (max initial / anyComponentStyle) | Ранний сигнал «раздулось» | + |
| Разово прогнать **`source-map-explorer`** или **`rollup-plugin-visualizer`** на `dist/.../browser/*.js` (без постоянного внедрения в репо) | Увидеть, кто занял место в чанке | + |
| **`rxjs`**: избегать тяжёлых операторов там, где хватит простых; не импортировать из внутренних путей | Чуть меньше и проще бандл | ~ |
| Проверить **`zone.js`**: не добавлять лишние `NgZone.run` без нужды | Меньше лишних тиков CD | ~ |

## 3. Runtime в приложении (без смены архитектуры)

| Сделать | Зачем | Эффект |
|--------|--------|--------|
| Убрать лишние **`subscribe()` без отписки** там, где долгоживущие компоненты (или `takeUntilDestroyed`) | Меньше утечек и лишней работы | ~ |
| **Тяжёлые списки**: виртуальный скролл там, где уже есть CDK — для очень длинных таблиц | Плавность скролла | ~ |
| **Change detection**: `OnPush` там, где дерево большое и нет лишних обновлений | Меньше проверок шаблона | ~ |
| Не держать **огромные массивы в сигналах/сторах** без нужды; пагинация с бэка где возможно | Меньше памяти в браузере | ~ |
| **Изображения в таблицах КП/витрины**: `loading="lazy"` где уместно (уже частично может быть) | Меньше конкуренции за сеть на первом экране | ~ |
| Проверить **дублирующиеся HTTP-запросы** (Network) на открытии хаба | Один запрос вместо двух | + |
| **`trackBy`** в больших `@for` / `*ngFor` где списки часто пересоздаются | Меньше DOM-операций | ~ |
| Не вызывать **тяжёлые функции из шаблона** без `pipe` с `pure: true` или предвычисления | Меньше работы на каждый CD | + |
| Проверить **`MatDialog` / порталы**: закрытие и уничтожение при уходе со страницы | Меньше висящих подписок и DOM | ~ |

## 5. Docker и деплой (уже существующий путь)

| Сделать | Зачем | Эффект |
|--------|--------|--------|
| Слои **`npm ci`** кэшировать (order COPY package*.json до копии исходников) — так в `Dockerfile.web.in-docker-build` | Повторные полные Docker-сборки быстрее | + |

| На сервере не запускать **`npm ci`** на хосте без нужды — только внутри образа | Меньше диска и путаницы | ~ |

| Сделать | Зачем | Эффект |

## 7. Что сознательно не делаем в этом чеклисте (по вашему запросу)

- **SSR / SSG / Angular Universal** — не предлагаем как шаг.
- **Смена бандлера или Nx → другой оркестратор** — вне scope.
- **Микрофронтенды** — вне scope.
- **Переезд API** — вне scope; только клиентская и сборочная оптимизация.
- **Service Worker / PWA / Workbox** — не добавляем «ради скорости» в этом проходе.
- **HTTP/3, Brotli на CDN** — зона инфраструктуры вне репо; при появлении nginx перед приложением — включить сжатие статики там.

При появлении отдельного решения на эти темы чеклист можно расширить отдельным документом.

## 8. Порядок работ «глобальной проверки» (рекомендуемый)

1. Замеры (раздел 0) — 1–2 часа с записью цифр в задачу или wiki.  
2. Самый жирный чанк из лога сборки + **lazy для тяжёлых страниц и опциональных библиотек** (раздел 2) — часто лучший **+/~** по труду.  
3. Docker context + кэш слоёв (разделы 1 и 5) — для сервера.  
4. OnPush и дубли запросов (раздел 3) — точечно по профилю.  
5. Чистка deps и мёртвого кода (раздел 4) — фоном, после стабилизации фич.

## 9. Краткая шпаргалка команд (копирование)

```bash
# из crm-web
npx nx build crm-web --verbose

# кэш Nx отключить для эксперимента
set NX_SKIP_NX_CACHE=true   # Windows cmd
$env:NX_SKIP_NX_CACHE='true' # PowerShell

# граф зависимостей
npx nx graph

# Docker из корня репо (как в deploy)
docker compose -f deploy/docker-compose.yml --env-file deploy/.env build web
```

## 10. Риски и ожидания

- **Минус времени сборки** на VPS без **RAM/swap** часто невозможен — сначала инфраструктура, потом микрооптимизации кода.  
- **Агрессивный lazy** без тестов маршрутов даёт **−** в виде регрессий навигации.  
- **Обновление major** Angular/Nx ради скорости даёт **−** по стоимости миграции — выносить в отдельный релиз.
- **Оптимизация «в ноль»** редко окупается: достаточно убрать 1–2 главных узких места из замеров.

## 11. Мини-чеклист перед релизом (быстрый)

- [ ] `nx build crm-web` проходит локально за приемлемое время.  
- [ ] В логе сборки нет неожиданного гигантского чанка (сравнить с прошлым релизом).  
- [ ] Lighthouse Performance на логине и одной тяжёлой странице не хуже прошлого снимка (или записать регрессию в задачу).  
- [ ] Docker `build web` на стенде не падает по OOM; при падении — лог с `--verbose`.  
- [ ] Нет новых огромных файлов в `public/` без обоснования.
