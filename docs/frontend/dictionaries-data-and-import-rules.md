# Данные справочников: нормализация, импорт/экспорт, аудит

**Цель:** единые правила, чтобы не плодить дубликаты («Алюминий» / «алюминий» / лишние пробелы), предсказуемо сопоставлять Excel с справочниками и чтобы следующая команда могла **пройти все таблицы по чеклисту** до следующего крупного этапа.

**Связанные документы:** поведение полей и Excel-канон — [`dictionary-field-behavior-guide.md`](./dictionary-field-behavior-guide.md); подписи UI — [`dictionaries-naming-convention.md`](./dictionaries-naming-convention.md); целевая модель материалов — [`../domain/material-and-material-characteristics.md`](../domain/material-and-material-characteristics.md); роли, маршруты и страница «Настройки» — [`rbac-and-admin-settings.md`](./rbac-and-admin-settings.md); справочник пользователей (в т.ч. Excel) — [`users-dictionary.md`](./users-dictionary.md).

---

## 1. Идеология хранения (коротко)

- **Мелкие справочники** — источник выбора в UI и при импорте.
- **Материалы (целевая таблица в Postgres):** храним **и ссылки на мелкие строки**, и **снимок подписей/цифр** на момент сохранения — чтобы списки и Excel не зависели от того, что завтра случится с мелкой записью. Подробности смысла — в доменном документе по материалам.

---

## 2. «Как называется» то, что вы хотите (без жаргона)

**Нормализация текста** — это не одна кнопка «сделать всё маленькими буквами», а **набор согласованных правил**: что мы убираем (пробелы по краям), как сравниваем две строки «одно и то же или нет», и **как именно сохраняем** в базе (что показываем человеку).

Рекомендуемый каркас (детали по типам полей — в чеклисте справочника):

| Шаг | Зачем |
|-----|--------|
| Убрать пробелы в начале и конце | Импорт из Excel и копипаст не создают «другую» запись из‑за пробела |
| При необходимости привести **коды** (артикулы, RAL, внутренние коды) к **одному регистру** (часто верхний) | Совпадение при импорте и отсутствие дублей `abc` / `ABC` |
| Для **наименований на русском** отдельно договориться: храним **как принято в отрасли** или сравниваем **без учёта регистра**, но не ломаем осмысленные заглавные (бренды, аббревиатуры) |
| Одно правило на весь справочник | Команда не гадает построчно |

**Импорт без id в файле:** в колонках — **человеческие значения**; приложение ищет строку в мелком справочнике по **тому же правилу нормализации**, что и при сохранении. Не нашлось / нашлось две — строка отчёта с ошибкой (без тихого пропуска), если иное не согласовано отдельно.

**Экспорт:** как в каноне — **только бизнес-колонки**; заголовки совпадают с шаблоном и валидатором импорта.

## 3. Правки мелких справочников и материалы

Уже согласованная логика уровня продукта:

- При изменении мелкой записи можно предлагать: **обновить снимки во всех связанных материалах** или **не трогать материалы**.
- Вариант «новая версия с новым id» в мелкой таблице оставляет старые материалы на старой связи и старых снимках — история не ломается.

Детали UX и API фиксируются при реализации; этот файл задаёт **ожидания по данным**, не экраны.

## 4. Чеклист аудита для каждого справочника (и для «Материалов»)

Перед тем как считать справочник «приведённым к новым правилам», заполнить для него блок:

1. **Уникальность:** по каким полям запрещены дубликаты после нормализации (например код + организация)?
2. **Нормализация по полям:** для каждого текстового поля — trim / регистр кодов / сравнение для импорта.
3. **Excel:** список колонок шаблона; импорт create-only или будущий upsert — как в [`dictionary-field-behavior-guide.md`](./dictionary-field-behavior-guide.md).
4. **Связи:** какие другие справочники выбираются селектами; как при импорте разрешаются текстовые колонки.
5. **Материалы (когда модель готова):** какие колонки — ссылки + снимки; отчёт об ошибках при неоднозначном совпадении.

Итог аудита: таблица «справочник → статус (ok / в работе)» в журнале релиза или в тикете — на усмотрение команды.

## 5. Порядок работ для команды (рекомендация)

1. Утвердить этот документ и дополнить примеры по **одному** эталонному справочнику (например единицы или цвета).
2. Реализовать общую утилиту нормализации в коде (один вход для форм и импорта).
3. Пройти существующие справочники на `/dictionaries` по чеклисту сверху вниз.
4. Затем миграция материалов по [`../domain/material-and-material-characteristics.md`](../domain/material-and-material-characteristics.md) и новые правила Excel для материалов.

## 6. Витрина правил для администратора (не «только под капотом»)

Задача: админ **видит** таблицу «поле → как вводить → что сделает импорт» (пробелы, регистр, слитно/через дефис, пример), без чтения кода.

### Проверенная идея: один источник правды

- Правила описываются **один раз** в машиночитаемом виде (лучше начать с **`*.json` в `assets/` или константы в `*.ts`** в репозитории — версионируется вместе с кодом).
- **Импорт/формы** вызывают общую функцию: «поле `units.code` → применить профиль `codeUpperTrim`».
- **Экран «Админ-настройки»** (`/settings`, блок правил полей) **только читает** тот же список и рисует таблицу: колонки вроде *Справочник*, *Поле*, *Подпись для людей*, *Правило*, *Пример до/после*, *Заметка для Excel*.
- Тогда админ и разработчик не расходятся: поменяли JSON/TS — обновилась и логика, и витрина (после деплоя).

### Ступени по сложности (рекомендация)

1. **Минимум (часто достаточно):** одна **read-only** страница в настройках + таблица из дизайн-системы (`content-card` + разметка как у других списков). Никакого редактора правил в UI на первом этапе.
2. **Средний уровень:** кнопка «Скачать описание правил» (PDF/Markdown), если бухгалтерии удобнее файл.
3. **Тяжёлый уровень (позже, если понадобится):** правила в **БД** и экран редактирования — оправдано только при необходимости менять правила **без релиза**; дороже в тестах и миграциях.

### Как не превратить Angular в кашу

- Отдельный небольшой блок: `features/settings/data-rules/` (или `admin/data-dictionary/`) — **один** сервис `FieldRulesCatalog`, **один** «тупой» компонент таблицы, типы правил в `model/`.
- Подписи для «большие/маленькие буквы», «слитно / через дефис» — это **не отдельные виджеты**, а **значения enum** + **pipe** или маленькая функция «иконка/бейдж для типа правила».
- Не смешивать с `dictionaries-page`: там уже тяжело; витрина правил — отдельный маршрут.

### Что делают другие продукты по смыслу

- **Каталог полей (data dictionary)** для сотрудников.
- **Описания полей в JSON Schema** — текст `description` показывают в админках.
- Идея та же: **метаданные полей** живут рядом с валидацией, UI лишь отображает.
