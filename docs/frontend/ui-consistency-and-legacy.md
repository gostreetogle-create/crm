# Единообразие UI: наследие, массовые правки и «что делать, если в доках не сказано»

## Канон (куда смотреть перед кодом)

1. **Поведение и паттерны:** [`interaction-system.md`](./interaction-system.md), [`development-workflow.md`](./development-workflow.md).
2. **Визуал и токены:** [`design-system.md`](./design-system.md), пиксель-чеклист [`visual-consistency-checklist.md`](./visual-consistency-checklist.md).
3. **Реализация общих компонентов:** библиотека **`@srm/ui-kit`** (`crm-web/libs/ui-kit/`). **Тема (токены, пресеты, store):** **`@srm/theme-core`** (`crm-web/libs/theme-core/`). Новый экранный UI по возможности собирается **только** из `ui-kit` + тема (`var(--...)`), без локальных палитр в фичах — как в [`development-workflow.md`](./development-workflow.md).

Если правило из канона **невозможно** соблюсти без компромисса — фиксируем отклонение в [`temporary-deviations-log.md`](./temporary-deviations-log.md) **до или вместе с** merge, со сроком/триггером выравнивания.

## Почему нельзя «сразу пробежаться по всему и поменять везде» одним махом

Это не запрет навсегда, а ограничение **безопасного** изменения:

| Фактор | Смысл |
|--------|--------|
| **Регрессии** | Сотни файлов → высокий риск сломать сценарии, доступность, мобильную вёрстку, Excel/импорт, права. |
| **Скрытые связи** | Локальные `scss`, селекторы, тесты, порядок слоёв модалок — не всегда видны из одного файла. |
| **Смысл vs внешний вид** | «Похоже» не всегда «одинаковый компонент»: иногда нужна отдельная вариация в `ui-kit`, а не копипаст. |
| **Слияния и ревью** | Огромный diff плохо ревьюится; ошибки пролезают. |
| **Приоритет** | Выгоднее идти **по зонам** (хаб справочников → настройки → …) с проверкой после каждой зоны. |

Итог: **массовое выравнивание делается планом** (разделы/эпики, ограниченные PR, сборка и ручной смоук по затронутым маршрутам), а не одной «невидимой» перекраской всего репозитория.

## Откуда берутся «разные элементы», если в доках написано «всегда так»

- **Наследие:** экраны и стили появлялись в разное время; документы описывают **целевое** состояние, но не откатывают историю автоматически.
- **Новый код в обход канона** — ошибка процесса: такой PR **не должен** проходить без приведения к `@srm/ui-kit` / токенам или без записи в `temporary-deviations-log.md`.
- **Пробел в доках** — не оправдание для произвольной вёрстки: **сначала уточнение / короткое предложение в канон** (или запись в отклонениях), **потом** реализация.

## Если в документации не хватает конкретики

1. **Не угадывать** уникальный UI «от себя» на всю фичу.
2. Ориентиры по умолчанию: **демо** (`/demo` / ui-demo), **эталонные** блоки из `@srm/ui-kit`, существующий ближайший экран в том же приложении.
3. Зафиксировать вопрос в задаче/PR и при необходимости **дописать один абзац** в `design-system.md` или `development-workflow.md` (согласование с владельцем канона по правилам репозитория).

## Мини-чеклист для исполнителя (и для ревью)

- [ ] Импорты UI из **`@srm/ui-kit`** там, где есть готовый примитив.
- [ ] Цвета/отступы через **токены темы**, не произвольный hex/px там, где есть токен.
- [ ] Повторяющийся паттерн **не копируется** второй раз — вынос в `ui-kit` или общий фрагмент.
- [ ] Временное отличие от канона — есть строка в **`temporary-deviations-log.md`**.

Обновление статуса «насколько выровняли продукт» — в [`dictionaries-polish-backlog.md`](./dictionaries-polish-backlog.md) и связанных чеклистах; это **учёт работ**, а не автоматический аудит каждого пикселя.
