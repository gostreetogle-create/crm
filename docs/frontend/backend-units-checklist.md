# Backend Checklist: Units API (`Единицы измерения`)

Цель: подключить frontend-справочник `Единицы измерения` к реальному API без доработок UI.

## 1) CRUD endpoints

- [ ] `GET /units` -> список единиц (`id, name, code, notes, isActive`)
- [ ] `POST /units` -> создание единицы
- [ ] `PUT /units/{id}` -> обновление единицы
- [ ] `DELETE /units/{id}` -> удаление единицы

## 2) Валидация на API

- [ ] `name` обязателен, длина >= 1
- [ ] `code` обязателен, длина >= 2
- [ ] `code` уникален в рамках tenant
- [ ] `notes` опционален
- [ ] `isActive` обязателен (boolean)

## 3) Совместимость с frontend

- [ ] JSON-поля совпадают с `UnitItem` в `features/units/model/unit-item.ts`
- [ ] `GET /units` возвращает массив (не объект-обертку)
- [ ] `POST/PUT` возвращают созданную/обновленную сущность
- [ ] Ошибки валидации возвращаются в предсказуемом формате (`field`, `message`)
- [ ] Терминология согласована: в API/доках backend использовать `units` / `единицы измерения`
- [ ] UI-сокращение `Ед. изм.` не используется как имя endpoint/DTO-поля
- [ ] Mapping UI->API подтвержден:
  - `Ед. изм.` -> `name`
  - `Код` -> `code`
  - `Комментарий` -> `notes`
  - `Активна` -> `isActive`

## 4) Массовый импорт (JSON, Admin API)

- [x] `POST /api/admin/bulk/units` — тело `{ items: [...] }`, до 5000 строк, ответ `{ ok, created, errors }` (пилот в репозитории)
- [ ] при необходимости: расширить контракт (upsert по `id`, dry-run) — отдельной спецификацией
- [ ] отчёт по ошибкам с **индексом** элемента в `items` и текстом причины

## 5) Нефункциональные требования

- [ ] аудит изменений справочника (кто/когда/что)
- [ ] пагинация/фильтрация по `name` и `code` (если объем вырастет)
- [ ] единые коды ошибок для frontend
