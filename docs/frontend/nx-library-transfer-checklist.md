# Nx: чек-лист по переносу в библиотеки

Цель: переносить код в библиотеки поэтапно, без поломок и лишней сложности.

## 0) Перед стартом

- [ ] Ветка создана от актуального `main`.
- [ ] Текущий `main` в рабочем состоянии (`serve`, `build`, `lint` проходят).
- [ ] Есть список кандидатов на перенос (не "все подряд", а 1-2 блока).
- [ ] Выбран владелец PR и формат ревью.

## 1) Что переносим в первую очередь

Переносить в библиотеки в таком приоритете:

1. **`libs/ui-kit`** (`@srm/ui-kit`) — переиспользуемые компоненты без доменной привязки (актуальный канон; старый каталог `src/app/shared/ui` не наращивать).
2. `@srm/shared-types` (типы, интерфейсы, контракты; не копить новое в `src/app/shared/model`).
3. `shared/utils` (чистые функции, форматтеры, мапперы).
4. `data-access` (API-клиенты, репозитории, адаптеры) - только после стабилизации UI.

Не переносить сразу:

- `features/*/pages/*` (страницы и orchestration-логика) - оставить в app.
- Сырые/часто меняющиеся куски, где требования еще "плавают".

## 2) Правила отбора кандидата в lib

Кандидат подходит, если выполнено хотя бы 2 условия:

- [ ] Используется в 2+ местах (или точно будет использоваться в другом app).
- [ ] Не зависит напрямую от конкретной страницы.
- [ ] Имеет понятный публичный API (`index.ts`, экспорт ограничен).
- [ ] Можно покрыть тестом отдельно от страницы.

Если условие не выполняется, оставить код в приложении до следующей итерации.

## 3) Минимальная итерация переноса (без перегруза)

- [ ] Создать 1 библиотеку под один слой (например `shared-ui`).
- [ ] Перенести 1-2 компонента/модуля (не больше за итерацию).
- [ ] Обновить импорты в приложении.
- [ ] Удалить дубли из старого места.
- [ ] Прогнать "роботов" (см. раздел 5).
- [ ] Открыть PR с коротким diff и понятным риском.

## 4) Структурные ограничения (чтобы не запутаться)

- [ ] У каждой библиотеки один явный слой ответственности.
- [ ] Публичные импорты только через `index.ts` (barrel).
- [ ] Нет циклических зависимостей между библиотеками.
- [ ] UI-библиотеки не тянут доменную бизнес-логику.
- [ ] `data-access` не импортирует UI.

## 5) "Проверка роботами" перед PR

Базовый набор:

- [ ] `npx nx lint crm-web`
- [ ] `npx nx test crm-web` (если есть/настроено)
- [ ] `npx nx build crm-web --configuration=development`

Если библиотек станет несколько:

- [ ] `npx nx affected -t lint,test,build`
- [ ] Проверка импорта по границам слоев (eslint module boundaries/tags).

## 6) Стоп-критерии (когда не продолжаем перенос)

Остановиться и зафиксировать промежуточный PR, если:

- [ ] Появились циклы зависимостей.
- [ ] Выросло число "временных костылей" в импортах.
- [ ] Растет diff без прироста переиспользования.
- [ ] Требуется массовый рефактор страниц для одного маленького выноса.

## 7) Практичный план именно для текущего проекта

Рекомендуемая последовательность:

1. Итерация A (выполнена): `crud-layout` (+ меню действий), `page-header`, `ui-modal`, `app-ui-button` в `libs/ui-kit` (`@srm/ui-kit`); демо в `srm-front` (`crud-smoke`). Детали: `srm-front-development-workflow.md` §8.
2. Итерация B (выполнена): `FieldRow` и контракт `DictionaryPropagation*` в `@srm/shared-types`; импорты в data-слое обновлены; см. `srm-front-development-workflow.md` §9.
3. Итерация C (выполнена): `API_CONFIG` в `@srm/platform-core`; вынесены **colors**, **materials**, **units**, **coatings**, **surface-finishes**, **geometries**, **material-characteristics**, **production-work-types**, **clients**, **organizations**, **roles**, **users** (`@srm/*-data-access`).
4. Итерация D (фундамент): правило ESLint `type:data-access` и зависимости app → data-access; далее — перенос остальных репозиториев по одному. Страницы `features/*/pages/*` пока в `crm-web`.
5. Итерация E (выполнена, волна 1..5): `@srm/theme-core`, `@srm/dictionaries-state`, `@srm/dictionaries-utils`, `@srm/settings-core`; в app оставлены shim-реэкспорты на новые библиотеки для безопасной миграции без массовых правок потребителей.
6. Итерация F (выполнена): удалён shim-слой в `src/app` (core/auth/api shim-файлы, feature state/utils shim-файлы, shared ui shim public-api), потребители переведены на прямые импорты `@srm/*`.

## 8) Definition of Done для каждой итерации

- [ ] Нет изменений в поведении UI (визуально и по сценариям).
- [ ] Все импорты из нового публичного API библиотеки.
- [ ] Линт/сборка/тесты проходят.
- [ ] PR содержит короткое описание: "что вынесли", "зачем", "какой риск".
- [ ] Следующая итерация определена (маленькая и измеримая).
