
## Уже зафиксировано в текущем `crm` (синхез чатов)

### Первый поток справочников manufacturing

- Внедрение **по одному пункту** с подтверждением; фронт на mock → затем REST.
- Порядок до материалов: **Вид работ** → единицы → цвет → отделка → геометрия → покрытие → **материалы последними**.

### Вид работ (справочник операций)

- Классификатор для **маршрута / Гантта**; назначение исполнителя — на **экземпляр задачи**, не на строку справочника.
- **Часы** по конкретной детали/заказу — в маршруте/задаче, не в карточке вида работ.
- **Ставка труда (`hourlyRateRub`, руб/ч)** хранится **в справочнике** по виду работ (например сварка 600 ₽/ч): это базовая учётная цена часа для расчёта трудозатрат. **Стоимость труда по операции** = часы (из маршрута) × ставка вида работ; при необходимости на заказе/КП — **снимок** ставки/часов по общим правилам снимков.
- Поля: `Наименование`, `Короткое_обозначение`, ставка руб/ч, `isActive`; системный ключ `id`. Отдельного бизнес-кода и отдельного поля «тип/группа операций» нет — процесс задаётся **именем** вида работ (лазер, сварка, покраска…). Уникальность на уровне наименования (согласовать с БД: UNIQUE на `name` или эквивалент).
- Связь с деталью/продуктом — через технологию (операции со **ссылкой на id** вида работ).

См. также `docs/frontend/work-types-crud.md`.

### Упрощённое производство (масштаб «~20 человек», черновик)

**Контекст:** первый контур планирования и учёта труда без избыточной детализации.

**Принятые решения:**

- Планирование в духе **диаграммы Ганта** (задачи по времени с зависимостями).
- **Ответственный / исполнитель** фиксируется на **экземпляре задачи** в плане или заказе, не в строке справочника «Вид работ».
- Учёт труда и оплата — **почасовая**; объёмы времени в этом контуре — **целые часы** (без минут/секунд и дробной части часа в бизнес-правилах первой версии).

**Границы:** смены, табель с поминутной детализацией, интеграции с ЗУП — позже по необходимости.

### Данные, версии, импорт, заказы

- Суррогатный PK; где нужен стабильный внешний ключ для людей/Excel — **бизнес-код** (в других справочниках); **вид работ** — исключение: только `id` + уникальное наименование. FK по `id`.
- `row_version` на **Деталь / Изделие / Продукт**; в справочниках — `is_active`.
- `revision` — инженерная метка, не SCD2.
- **КП / заказ:** снимок строк; изменение справочника не переписывает прошлые сделки.
- Excel: только бизнес-колонки; импорт create-only без системных полей до upsert.

### Организации и контакты для КП (2026-03-30)

- Вводим отдельный справочник **«Организации»** (юридические лица) для шапки КП и договорных реквизитов.
- Справочник **«Контактные лица»** хранит физлиц; организация может иметь **несколько** контактов.
- Связи в UI задаются через множественный выбор `contactIds`; при сохранении КП в документ уходит снимок нужных данных.
- Для связанных полей на формах используем единый паттерн: `select` + квадратная кнопка `+` рядом для quick-add записи в связанном справочнике.

### Шапка коммерческого предложения (2026-03-30)

- В БД: модель **`CommercialOffer`** — документ верхнего уровня (не справочник хаба): номер/название, статус жизненного цикла, опциональные ссылки на **организацию** (от чьего имени) и **контактное лицо** (контрагент), срок действия, валюта, примечание.
- Статусы: `DRAFT` → `SENT` → `ACCEPTED` / `CANCELLED` (перечисление в Prisma).
- REST: `GET/POST/PUT/DELETE` под префиксом `/api/commercial-offers` (аутентификация как у остальных сущностей). В ответе дублируются человекочитаемые подписи `organizationLabel` и `clientLabel` для списков.
- **Строки КП (позиции, снимки цен)** — отдельная таблица и этап согласования; до этого шапка может существовать без суммы по строкам.

### Калькуляция и цены (черновик в `crm`)

- Разделять **себестоимость** и **цену продажи**; материал → затраты; **труд** → для каждой операции маршрута: норма/факт **часов** × **ставка ₽/ч** из справочника «Вид работ» (`hourlyRateRub`, **не меньше 1**); итог по продукту/заказу суммированием.
- **Материалы:** в карточке — **`purchasePriceRub`** (₽) и **`unitId`** из справочника единиц; смысл цены = **за одну эту единицу**. Цвет/покрытие задают контекст, не отдельный прайс RAL — см. `docs/frontend/costing-and-dictionary-prices.md`.
- **Целевое разнесение (ещё не в коде):** справочник **«Характеристики материала»** (профиль: плотность, цвет, финиш, покрытие …) отдельно от учётной строки **«Материал»** (ссылка на характеристику + **форма и габариты** + цена/единица). См. [`docs/domain/material-and-material-characteristics.md`](domain/material-and-material-characteristics.md).

### UI справочников

- Таблица: короткие `label`; модалка: полные подписи; TS/`key` стабильны.
- Порядок **карточек** на `/dictionaries`: **материалы сверху** (по запросу UX), затем вид работ → единицы → цвет → отделка → геометрия → покрытие. Логика внедрения по-прежнему «материалы после опорных справочников» — при первом заполнении сначала заведите единицы/цвета и т.д., затем материал. После введения **характеристик материала** порядок уточнить: опорные справочники → характеристики → геометрия → **материалы** (позиции).

## Наследие из `crmgenerator_nx` (объединённая модель)

Источник: [BUSINESS_LOGIC.md](https://github.com/gostreetogle-create/crmgenerator_nx/blob/main/docs/business/BUSINESS_LOGIC.md), [PRICING_AND_MARKUP_MODEL.md](https://github.com/gostreetogle-create/crmgenerator_nx/blob/main/docs/business/PRICING_AND_MARKUP_MODEL.md).  
**Имеет силу как целевая предметная модель на будущее**, пока в `crm` нет Prisma/API под КП и заказы — сверяться при проектировании.

### Роли домена (краткий глоссарий)

| Идея в архивном проекте | Смысл | Сопоставление с `crm` / manufacturing JSON |
|-------------------------|--------|-----------------------------------------------|
| `WorkKind` | Вид работ (лазер, сварка, …) | ≈ `Справочник_Вид_работ` / `production-work-types` |
| `ProductWorkKind` | Привязка вида работ к **товару** для производства | Позже: строки технологии на продукте/узле |
| `OperationAssignment` | Вид работ + **исполнитель** на строке заказа | План/заказ, не справочник |
| `Product` | Логический товар каталога | ≈ сущность `Продукт` в JSON |
| `ProductSpecification` | Строка **состава** каталога (тип детали + опционально материал) | Родственно BOM / `Состав_*` в JSON; детализация при согласовании |
| `Proposal` + `ProposalItem` | КП и **строки-снимки** | Согласуется с нашим правилом «снимок в КП» |
| `Order`, `OrderLine` | Заказ и строки | В архиве + snapshot связи `proposalId` / `proposalItemId` |
| `Warehouse`, `Stock`, `StockMovement` | Склад, остатки, движения | Целевая зона; в текущем `crm` не реализовано |

### Прогрессивное заполнение товара и готовность (Readiness)

- Товар **не обязан** быть полным сразу: этапы КП → снабжение → производство с разным набором обязательных полей (как в архивном документе).
- Индикатор **готовности** (базовый / материалы / виды работ / полный) логично считать **на лету**, не дублировать «флажками» в БД без нужды.
- Предупреждение при КП без состава: себестоимость может быть 0 — **не обязательно блокировать**, если политика такая.

### Ценообразование и строка КП (расширение черновика)

Цепочка из архивного проекта (для будущей реализации):

- `Product.listPrice` — базовая **цена продажи** (ручной ввод).
- `Product.costPrice` — **себестоимость**, считает сервер (материалы + труд по согласованной формуле).
- `Material.listPrice` или учётная цена — участие в cost (в архиве — закупочная у материала).
- Наценки `Organization.markup`, `Client.clientMarkup`, скидки на документ и по строке, НДС, округление до целевой суммы (`roundingAmount` / `targetTotal`) — см. полную схему в PRICING в старом репо.
- В **строке КП** хранить **snapshot**: каталожная цена, проценты наценок, итоговая `unitPrice`, ставка НДС по строке, при необходимости скидка по строке — чтобы история не «поехала» при смене прайса.

Это **усиливает** наш принцип «строка КП не пересчитывается из каталога задним числом» и задаёт конкретные поля на будущее.

### Инварианты из архивного BL (переносим без изменений смысла)

- Отправленное/принятое КП: строки только из **снимка**, не живой каталог.
- Связь производства с КП: хранить идентификаторы КП и строки (`proposalId`, `proposalItemId` или эквивалент).
- На строке/снимке заказа (архив: `OrderLine`): хранить **snapshot товара/комплектации** на момент создания для восстановления состава (гарантийный/юридический сценарий).
- Удаление справочников с активными ссылками: **деактивация**, не обязательный hard delete.

### Открытые вопросы из архива (ещё не закрыты в `crm`)

- Одно КП → один или несколько заказов.
- `Performer` = пользователь или отдельный справочник исполнителей.
- Гранулярность склада (партия/ячейка).
- Версии КП и freeze после оплаты — см. lifecycle в старом репо (`WORKFLOW_STATUSES` и т.д.) при переносе.

## Сведение: архив vs текущий `crm`

| Тема | `crmgenerator_nx` | Текущий `crm` |
|------|-------------------|----------------|
| Каталог/товар | Product + ProductSpecification + variants legacy | Manufacturing JSON (`Продукт`, составы); UI `/dictionaries` пока без сущности КП |
| Вид работ | `WorkKind` | `production-work-types` на фронте |
| Backend | Express + Prisma + PostgreSQL | Пока mock, Units с опциональным HTTP |
| КП/заказ/склад | Описано в BL и контракте | Шапка КП в Prisma + API; строки и заказ — позже |

**Правило при конфликте:** приоритет явно согласованного поведения в **текущем** репозитории и в чате; архив — источник **дополнений** до тех пор, пока не заменён актуальной спецификацией.

### Роли и доступ к CRM (имитация до API) (2026-03-29, уточнено 2026-03-29)

**Контекст:** в продукте нужны настраиваемые роли и права на разделы, плитки хаба справочников, действия CRUD и Excel.

**Принятые решения:** стабильные строковые ключи `PermissionKey` в коде. Роли — **записи справочника** на фронте (до API — mock + `localStorage` для матрицы). **Один суперадминистратор** (`isSystem`): полный набор прав, не настраивается матрицей и не показывается в колонках «Админ-настроек». Остальные роли: по умолчанию без прав, пока не заданы в матрице в браузере. Ключи `dict.hub.*` управляют видимостью плиток на `/dictionaries`. Канон по умолчанию для кодов ролей в коде — только `admin` (полный доступ); см. `DEFAULT_ROLE_PERMISSIONS_BY_CODE` в `authz.matrix.ts`. Подробности и чеклисты: `docs/frontend/rbac-and-admin-settings.md`.

**Открыто:** привязка ролей к пользователям и организациям на сервере; аудит изменений прав; вынос контракта ролей из `features` в `core` при необходимости.
