# Руководство по изделию (Product): поля, состав, расширения

**Область документа:** производственное изделие (**`ManufacturedProduct`** в Prisma, таблица **`"Product"`**, плитка «Изделия», API `/api/products`). Каталожные товары в составе **комплекса** (`Product` → таблица `products`) описаны отдельно: [`catalog-suite-guide.md`](./catalog-suite-guide.md).

Документ для продукта и аналитики: **что уже заложено в CRM**, **что вы перечислили как стандарт**, **чего нет в базе** и **как лучше сделать состав «изделие из изделий»**.

Техническая реализация хаба и API: [`docs/frontend/products-dictionary.md`](../frontend/products-dictionary.md).

### Реализовано (2026-04-04)

- **`code`** — артикул / внутренний код SKU на шапке `Product`, опционально, **уникален среди непустых** значений (индекс в БД).
- **`description`** — текст для витрины и КП; **`notes`** остаётся **внутренними заметками**.
- Хаб: колонки «Артикул», «Описание» в полной таблице; в свёрнутой плитке подпись строки — `артикул — наименование` или только наименование.
- Форма и просмотр в модалке изделия обновлены под эти поля.

## 1. Что есть в системе сейчас (канон кода и БД)

Модель **`Product`** (справочник «Изделия») — это **шапка** + **несколько строк состава** (`ProductLine`), каждая строка ссылается на **деталь производства** (`ProductionDetail`), а не на другое изделие.

### Шапка изделия

| Поле (смысл) | В API / БД | Обязательность | Примечание |
|----------------|------------|----------------|------------|
| Артикул | `code` | нет | Уникален, если задан; для списков и учёта SKU. |
| Наименование | `name` | да | Краткое имя изделия для списков и КП. |
| Описание (витрина / КП) | `description` | нет | Текст для клиента; без отдельного rich-text в модели. |
| Цена, ₽ | `priceRub` | нет | Может быть пусто — тогда на бэкенде может подставляться сумма по строкам состава (логика согласована с формой). |
| Себестоимость, ₽ | `costRub` | нет | То же про подстановку снизу по строкам. |
| Заметки (внутренние) | `notes` | нет | Свободный текст для операторов; не смешивать с `description`. |
| Активен | `isActive` | да | Участвует в выборках и хабе. |
| Служебные | `id`, `createdAt`, `updatedAt` | — | Не показываем в UI как бизнес-поля. |

### Строка состава (`ProductLine`)

| Поле (смысл) | В API / БД | Обязательность | Примечание |
|----------------|------------|----------------|------------|
| Деталь производства | `productionDetailId` | да | Ссылка на справочник «Детали» (оттуда тянутся материал, геометрия, исходные вид работ и т.д.). |
| Порядок строки | `sortOrder` | нет (дефолт 0) | Порядок в составе. |
| Вид работ (переопределение) | `workTypeId` | нет | Если пусто — берётся из детали/контекста детали. |
| Цвет (переопределение) | `colorId` | нет | Если пусто — из связанных справочников по детали/материалу. |

**Итог:** «Характеристики» в широком смысле сейчас **раскладываются по детали производства и материалу**, а не хранятся одной строкой на изделии. На шапке изделия есть **артикул** (`code`) и **описание для клиента** (`description`). **Фото**, **документы**, **длина/высота на шапке изделия** в модели `Product` **пока нет**.

## 2. Сверка с вашим «стандартным» списком

| Ваш пункт | Как закрыть сейчас | Статус |
|-----------|-------------------|--------|
| Артикул | `code` на `Product` (уникальность непустых). | **Есть** |
| Наименование | `name` | **Есть** |
| Фото | Нет; обычно `photoId` / галерея / ссылка на `kp_photos` или общий медиа-контур. | **Пробел** |
| Характеристики изделия | Частично через состав (детали) и связанные справочники; не единый блок «характеристики» на шапке. | **Частично** |
| Описание | `description` (витрина) и `notes` (внутреннее). | **Есть** |
| Цвет | На уровне **строки состава** (`colorId`), не обязательно одно поле «цвет изделия». | **Есть** (в составе) |
| Длина, высота | Не на `Product`; могут жить в **геометрии/детали**, если это параметры заготовки. | **Зависит от детали** |
| Документы | Нет сущности «документ изделия»; нужен контур вложений (файлы + тип + связь с `productId`). | **Пробел** |
| Себестоимость | `costRub` (+ агрегация по строкам при пустом значении) | **Есть** |
| Этап готовности | Не в модели; убираем из требований, как вы и сказали. | **Не делаем** |

Если «Ножен» / категория имелось в виду как **тип или группа изделий** — это отдельное поле `categoryId` / `productGroup` / теги; в текущей схеме **нет**.

## 3. Изделие состоит из изделий: как лучше сделать

Сейчас состав — только **«деталь производства» → строка**. Это не то же самое, что **«вложенное изделие»** (комплект, узел из готовых SKU).

### Рекомендация

1. **Целевая модель (надёжно для учёта и КП):** отдельная таблица связей **многие-ко-многим** между продуктами, например:
   - `ProductComponent` (или `ProductBundleLine`): `parentProductId`, `childProductId`, `quantity`, `sortOrder`, опционально `notes`.
   - Ограничения: запрет циклов (A → B → A) на уровне приложения или проверки; при необходимости — лимит глубины.
   - В UI: мультивыбор дочерних изделий в форме родителя **или** отдельная вкладка «Состав из изделий» с таблицей — как удобнее операторам.

2. **Временный UX без новой таблицы (хуже для учёта):** притворять «изделие» **деталью производства** с однозначным маппингом — быстро ломается, если у изделия своя цена/артикул/фото.

3. **«Потом отдельная таблица»** — как вы и предложили: это **правильный** путь для «несколько изделий из выпадающего списка». Мультиселект в форме — просто представление; источник правды — строки связи в БД.

Итог: **лучше заложить явную связь product–product** (с количеством и порядком), не хранить это только в UI.

## 4. Предлагаемый порядок внедрения расширений

1. ~~**Артикул** (`code`) на шапке + правила уникальности.~~ — **сделано** (2026-04-04).  
2. ~~**Описание для клиента** vs **внутренние заметки** — разнести поля.~~ — **сделано**: `description` и `notes`.  
3. **Фото** — один объект хранения файлов + ссылка с `Product`.
