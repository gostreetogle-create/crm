# Реестр manufacturing-справочников

## Назначение

Файл [`crm-manufacturing-dictionaries.json`](crm-manufacturing-dictionaries.json) — предметный реестр: ключ сущности, `status`, описание, список полей в терминах домена (ГОСТ/ЕСКД/ОКЕИ). Поле **`domainVsUi`** у записей со `status: "exists"` фиксирует расхождение между этим контуром и текущей моделью в `crm-web` (см. пути в тексте).

- **`exists`** — в приложении уже есть карточка в `/dictionaries` (или эквивалентный справочник).
- **`new`** — в UI ещё нет; реализация только после явного согласования по процессу ниже.

Черновики заметок и разбор «до реестра» можно складывать в **`справочники/tips/manufacturing/`** (папка под короткие материалы; канон данных остаётся в JSON).

## Нумерованный бэклог (`status: "new"`, нет карточки в `/dictionaries`)

Рекомендуемый порядок (от простого к зависимым; можно менять по приоритету бизнеса):

| № | Ключ в JSON | Комментарий |
|---|-------------|-------------|
| ~ | `Справочник_Вид_работ` | **Сделано** в UI (`exists`); первый пункт «первого потока» по фронту. |
| 1 | `Продукт` | Каталог/КП; логично перед составом продукта. |
| 2 | `Состав_продукта` | Зависит от `Продукт` и способа идентификации изделия. |
| 3 | `Деталь` | ЕСКД; зависит от материалов, единиц, покрытий, финиша, вида работ. |
| 4 | `Изделие` | Сборка; та же зона зависимостей. |
| 5 | `Состав_изделия` | Спецификация; после `Изделие`/`Деталь` и правил обозначений. |

**Уже в UI (`exists`), не в бэклоге «new»:** `Справочник_Цвет`, `Справочник_Вид_работ` и остальные карточки `/dictionaries` из JSON со статусом `exists`.

**Планируемое разнесение материалов:** текущая карточка «Материалы» по смыслу ближе к будущему справочнику **«Характеристики материала»**; отдельно появится учётная сущность **«Материал»** (характеристика + геометрия + цена/ед.). Контракт и карта полей: [`docs/domain/material-and-material-characteristics.md`](../docs/domain/material-and-material-characteristics.md).

## Шлюз согласования пункта N (перед кодом)

1. Выберите номер из таблицы бэклога (или согласуйте другой порядок).
2. Зафиксируйте для пункта **N** таблицу полей (минимум):

   | Доменное поле (из JSON) | Подпись в UI (RU) | Имя в TS/API | Тип | Обяз. | Примечание |
   |-------------------------|-------------------|--------------|-----|-------|------------|
   | … | … | … | string/number/bool/FK | да/нет | уникальность, формат |

3. Сверка с [`docs/frontend/dictionaries-naming-convention.md`](../docs/frontend/dictionaries-naming-convention.md) и чеклистом [`docs/frontend/new-dictionary-checklist.md`](../docs/frontend/new-dictionary-checklist.md).
4. Явное сообщение: **«ок, пункт N»** (или список правок полей → повтор шага 2).
5. Только после п.4 — ветка с кодом (mock store, `dictionaries-page`, права; массовый импорт — по `docs/dev-bulk-json-migration-checklist.md`, не на плитке).

Расширение полей у уже существующих карточек — **отдельная** задача: согласовать маппинг JSON ↔ текущий `*Item` и обновить `domainVsUi` в реестре.

Сводная **бизнес-логика** (вид работ, Гантт/маршрут, снимки заказа, калькуляция, задел реестра РФ и т.д.): [`docs/business-logic-canon.md`](../docs/business-logic-canon.md).
