## A. Видео / данные из Inventor → API → автозаполнение (в т.ч. заметки)

**Вопрос:** помешает ли это появлению новых полей?

- **Новые поля в CRM** всегда означают: модель TS/DTO, форма, при необходимости Excel, миграция БД, обновление `domainVsUi` в JSON.
- **Поток Inventor → API** добавляет слой **маппинга** (что отдаёт CAD/плагин ↔ какие поля пишем в CRM). Любое новое поле в CRM должно быть либо заполняемым из Inventor (если источник есть), либо явно «только ручной ввод».
- Поле **`notes`** (заметки) сейчас текстовое: его можно заполнять из API одной строкой (сводка из Inventor), не ломая остальную модель — при условии согласования формата (plain text vs структурированный JSON внутри строки).

### Чеклист перед внедрением Inventor/API

- [ ] Зафиксировать **минимальный набор** свойств, которые стабильно доступны из Inventor (имена свойств / iProperties).
- [ ] Решить **идемпотентность**: обновление по `id` детали/заготовки vs создание новой записи справочника.
- [ ] Согласовать **источник истины**: перезапись полей CRM из Inventor только по явной команде или по расписанию; конфликт с ручными правками.
- [ ] Для **`notes`**: лимит длины, кодировка, запрет опасных символов при необходимости; версионирование «что пришло из Inventor» (опционально отдельное поле `sourceNotes` позже).
- [ ] Спецификация **REST** (или очередь): эндпоинт, авторизация, идемпотентный ключ, ответы об ошибках по строкам при пакетной загрузке.
- [ ] Обновить **документацию контракта** (отдельный doc или раздел в backend-map, когда появится бэк).
- [ ] **Объект в CRM:** что именно создаём/обновляем — строка справочника «Форма и габариты», черновик **Деталь**, или только поля в заказе/КП (без новой сущности).
- [ ] **Частота:** разовый импорт по кнопке, фоновая синхронизация, webhook от плагина — влияет на очередь и блокировки.
- [ ] **Аудит:** логировать `source`, `externalId`, время последней синхронизации (минимум для разбора расхождений).
- [ ] **Ошибки:** что показываем пользователю при частичном успехе (часть свойств не пришла из Inventor).
- [ ] **Безопасность:** кто может вызывать API (роль, API key, сеть/VPN); лимиты размера тела запроса.

## B. Сейчас у каждого справочника свой Excel — как к одному файлу с листами (вкладками)?

**Короткий ответ:** текущий подход (отдельный `exportRowsToExcel` / свой парсер на справочник) **не мешает** позже собрать **один workbook с несколькими листами**: это обычно тонкая обёртка над уже существующими функциями «строки + заголовки» для каждой сущности.

**Когда понадобится рефакторинг, а не «лёгкий жест»:**

- если имена колонок / порядок на листах должны быть **жёстко унифицированы** под внешний стандарт (не как сейчас в UI);
- если нужен **единый импорт** «весь файл сразу» с **транзакцией** (все листы или ни один);
- если появятся **ссылки между листами** в одном файле (формулы, именованные диапазоны).

**Рекомендация заранее (низкая стоимость сейчас):**

- Держать для каждого справочника **одну функцию** вида «массив строк + список заголовков» (как сейчас по сути в `dictionaries-page`) без дублирования бизнес-логики в XLSX-слое.
- Имена листов согласовать с ключами из **`crm-manufacturing-dictionaries.json`** (латиница/транслит), чтобы «сводный экспорт» был предсказуемым.

### Чеклист перед «сводным Excel»

- [ ] Список листов и точное имя каждого листа (1:1 с карточками `/dictionaries`).
- [ ] Для каждого листа: те же колонки, что в одиночном экспорте, или осознанное отличие (задокументировать).
- [ ] Импорт: только экспорт (обратная совместимость) или также **multi-sheet import** и порядок применения (FK: сначала единицы, потом материалы и т.д.).
- [ ] Ошибки по листам: агрегировать в один отчёт пользователю.
- [ ] Права: одно действие «скачать всё» vs сохранить отдельные права по справочникам.
- [ ] **Имя файла и версия:** фиксированное имя (`manufacturing-dictionaries.xlsx`) или с датой; нужен ли манифест версии формата в первом листе или в `README` внутри архива.
- [ ] **Пустые листы:** экспортировать листы без данных или скрывать; импорт — игнорировать пустой лист или считать ошибкой.
- [ ] **Кодировка и локаль:** разделитель десятичных, даты (ISO vs RU) — единое правило для всех листов.
- [ ] **Совместимость:** одиночные файлы по справочникам остаются официальным путём для массового редактирования одной сущности, сводный — только отчёт/бэкап или полноценная замена — зафиксировать.

## C. Недосказанности, которые не блокируют текущий релиз фронта

Имеет смысл явно назвать на комитете, **чтобы потом не переделывать контракт**:

1. **Inventor:** формат обмена — только текстовые iProperties, или ещё STEP/масса из расчёта, или отдельный JSON от макроса; от этого зависит размер API и поля DTO.
2. **Идентификация детали:** внешний ключ (номер чертежа, Inventor `Document` id, hash файла) — без этого идемпотентность «обновить ту же строку» не определена.
3. **Сводный Excel и FK:** порядок листов при импорте должен совпадать с графом зависимостей (см. бэклог в [`справочники/README.md`](../../README.md)); иначе — только экспорт «на чтение» в v1.
4. **Геометрия в CRM:** компактная строка «Параметры» и тип формы — контракт для Excel и UI описан в [`docs/frontend/geometries-crud.md`](../../../docs/frontend/geometries-crud.md); расширение полей (например отдельный тип «прямоугольная труба») = согласование JSON + миграция + парсер Excel.
