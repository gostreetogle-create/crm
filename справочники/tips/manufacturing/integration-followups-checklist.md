Вернуться к этому файлу при планировании интеграций. Канон реестра полей — [`crm-manufacturing-dictionaries.json`](../../crm-manufacturing-dictionaries.json).

## A. Видео / данные из Inventor → API → автозаполнение (в т.ч. заметки)

**Вопрос:** помешает ли это появлению новых полей?

- **Новые поля в CRM** всегда означают: модель TS/DTO, форма, при необходимости поля в контракте массового импорта, миграция БД, обновление `domainVsUi` в JSON.
- **Поток Inventor → API** добавляет слой **маппинга** (что отдаёт CAD/плагин ↔ какие поля пишем в CRM). Любое новое поле в CRM должно быть либо заполняемым из Inventor (если источник есть), либо явно «только ручной ввод».
- Поле **`notes`** (заметки) сейчас текстовое: его можно заполнять из API одной строкой (сводка из Inventor), не ломая остальную модель — при условии согласования формата (plain text vs структурированный JSON внутри строки).

### Чеклист перед внедрением Inventor/API

- [ ] Зафиксировать **минимальный набор** свойств, которые стабильно доступны из Inventor (имена свойств / iProperties).
- [ ] Решить **идемпотентность**: обновление по `id` детали/заготовки vs создание новой записи справочника.
- [ ] Согласовать **источник истины**: перезапись полей CRM из Inventor только по явной команде или по расписанию; конфликт с ручными правками.
- [ ] Для **`notes`**: лимит длины, кодировка, запрет опасных символов при необходимости; версионирование «что пришло из Inventor» (опционально отдельное поле `sourceNotes` позже).
- [ ] Спецификация **REST** (или очередь): эндпоинт, авторизация, идемпотентный ключ, ответы об ошибках по элементам при пакетной загрузке.
- [ ] Обновить **документацию контракта** (отдельный doc или раздел в backend-map, когда появится бэк).
- [ ] **Объект в CRM:** что именно создаём/обновляем — строка справочника «Форма и габариты», черновик **Деталь**, или только поля в заказе/КП (без новой сущности).
- [ ] **Частота:** разовый импорт по кнопке, фоновая синхронизация, webhook от плагина — влияет на очередь и блокировки.
- [ ] **Аудит:** логировать `source`, `externalId`, время последней синхронизации (минимум для разбора расхождений).
- [ ] **Ошибки:** что показываем пользователю при частичном успехе (часть свойств не пришла из Inventor).
- [ ] **Безопасность:** кто может вызывать API (роль, API key, сеть/VPN); лимиты размера тела запроса.

## B. Несколько справочников: как к одному «пакету» без Excel в UI

**Короткий ответ:** на стороне CRM целевой канал — **несколько вызовов** `POST /api/admin/bulk/<сущность>` (или один агрегирующий endpoint, если согласуют), с **фиксированным порядком** применения по графу FK. Внешний обмен (Inventor, ETL) может отдавать один JSON с вложенными массивами — это маппится на те же bulk-операции на сервере.

**Когда нужен отдельный «сводный» контракт:**

- если внешняя система не может слать по одному справочнику;
- если нужна **одна транзакция** «всё или ничего» на несколько сущностей;
- если есть **ссылки между элементами** внутри одного пакета (порядок создания задаётся явно).

**Рекомендация:**

- Для каждого справочника держать **один** серверный обработчик bulk с ясным списком полей и отчётом ошибок по индексам (как пилот для `units`).
- Имена сущностей в пакете согласовать с ключами из **`crm-manufacturing-dictionaries.json`** (латиница/транслит), чтобы интеграции были предсказуемыми.

### Чеклист перед «сводным пакетом»

- [ ] Состав сущностей и порядок применения (граф FK).
- [ ] Для каждой сущности: поля payload, режим create-only vs upsert.
- [ ] Ошибки: агрегировать по сущностям и индексам; частичный успех — допустим или нет.
- [ ] Права: одно право «полный пакет» vs раздельные `admin.bulk.*`.
- [ ] Версия формата пакета (поле `version` в корне JSON или отдельный doc).
- [ ] Пустые массивы: игнорировать или считать ошибкой.
- [ ] Кодировка, числа, даты — единое правило для всего пакета.

## C. Недосказанности, которые не блокируют текущий релиз фронта

Имеет смысл явно назвать на комитете, **чтобы потом не переделывать контракт**:

1. **Inventor:** формат обмена — только текстовые iProperties, или ещё STEP/масса из расчёта, или отдельный JSON от макроса; от этого зависит размер API и поля DTO.
2. **Идентификация детали:** внешний ключ (номер чертежа, Inventor `Document` id, hash файла) — без этого идемпотентность «обновить ту же строку» не определена.
3. **Сводный пакет и FK:** порядок сущностей при применении должен совпадать с графом зависимостей (см. бэклог в [`справочники/README.md`](../../README.md)); иначе в v1 — только по одному справочнику за запрос.
4. **Геометрия в CRM:** компактная строка «Параметры» и тип формы — контракт для UI и для полей JSON описан в [`docs/frontend/geometries-crud.md`](../../../docs/frontend/geometries-crud.md); расширение полей (например отдельный тип «прямоугольная труба») = согласование JSON + миграция + парсер/валидация на сервере.
